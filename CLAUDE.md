## Commands

```bash
npm run dev          # Start dev server at http://localhost:3000
npm run build        # Production build (runs `prisma generate` first)
npm run lint         # ESLint
npm test             # Vitest unit tests (pure lib functions — no DB/API needed)
npx prisma generate  # Regenerate Prisma client after schema changes
git push origin main # Deploy to production (GitHub → Vercel auto-deploy)
```

### Schema changes (IMPORTANT — Turso gotcha)
`prisma db push` / `prisma migrate` do **NOT** work against Turso (`libsql://` → error P1013;
the schema engine only speaks `file:`/postgres). To apply a schema change:
1. Edit `prisma/schema.prisma`, then `npx prisma generate`.
2. Generate DDL: `npx prisma migrate diff --from-empty --to-schema prisma/schema.prisma --script`
   (note: the flag is `--to-schema`, not `--to-schema-datamodel`).
3. Run **only the new** `CREATE TABLE`/`ALTER` statements against Turso via the `@libsql/client`
   `executeMultiple()` (a tiny throwaway script that reads `DATABASE_URL`/`DATABASE_AUTH_TOKEN`
   from `.env`). Inspect data the same way, or with `turso db shell korean-study`.
   (`prisma studio` also can't connect to `libsql://`.)

## Architecture

**Stack:** Next.js 16 App Router, React 19, TypeScript, Tailwind CSS v4, Prisma 7 + libSQL (SQLite/Turso), Claude API (**claude-opus-4-8** extraction with adaptive thinking; **claude-haiku-4-5-20251001** for tap-to-gloss), Google Docs API v1, ElevenLabs / Google Neural2 TTS (`lib/tts.ts` swappable provider; active: ElevenLabs), Vercel Blob (TTS audio cache), lucide-react (nav icons), sharp (icon generation, devDep).

**Data flow:**
1. User triggers sync → `POST /api/sync` fetches the Google Doc via `lib/google-docs.ts` (reads **only the `수업 노트` tab**; now also captures bold/underline/highlighted spans as an `emphasized[]` list per lesson). Splits into lessons at horizontal rules, hashes each to detect new content. Sends lesson text + emphasized terms + existing deck's normalized fronts to Claude via `lib/extract-cards.ts`. **Exhaustive extraction** — every word, grammar pattern, and phrase becomes a card. Upserts cards by `normalizedFront` (real DB dedup). Processes **1 lesson per request** (`MAX_LESSONS_PER_SYNC = 1`) to stay under the 60 s Vercel function timeout. Reports `remaining` so the backlog drains over repeated sync taps. After all upserts, resolves each card's `components[]` lemmas → `CardDependency` edges (two-phase link). For **bulk re-extraction** (all lessons at once), use `npx tsx scripts/local-resync.mts` — runs locally with no timeout.
2. Study session → user opens `/study`, an async RSC (`app/study/page.tsx`) that calls `getStudyCards()` (`lib/study-cards.ts`) + `prisma.lesson.findMany()` server-side and passes the results as props to `components/StudyClient.tsx`, which starts directly in `'select-mode'` — no blank loading phase on first load. `getStudyCards()` returns cards where `CardReview.nextReview <= now`, optionally filtered by `lessonFrom`/`lessonTo` and `scope` (`due` | `ahead`). It fetches the whole eligible pool (up to 1000) and the "known lemmas" set (FSRS state ≥ 1) **concurrently via `Promise.allSettled`** — pool failure throws (500), known-lemmas failure degrades to an empty Set (non-critical ranking signal). Runs `selectSessionCards()` to pick a downward-closed set (prerequisites included, cap applied during selection), then `sequenceCards()` from `lib/sequence.ts` to reorder for foundation-first learning, then annotates every sentence with `unknownCount` via `lib/known-words.ts`, and serializes all Prisma `Date` fields to ISO strings (see `lib/dto.ts`) before returning. `GET /api/cards/due` delegates to the same `getStudyCards()` function (used by the lesson-range filter's client-side re-fetch). `StudySession` consumes the list **in server order** (no client shuffle). The flashcard Exposure front shows a bare Korean word when the card is new (state 0/1) AND the best sentence still has unknown context words; otherwise shows the sentence directly. When a sentence is shown, the one with fewest unknown words is chosen, tie-broken by `hashStr(id) + reps`. Grading a card is **synchronous and optimistic**: `submitReview` computes the FSRS result client-side via `reviewCard()` and advances the queue immediately; `POST /api/review` fires in the background (`.catch(() => {})`, not awaited) — no perceptible delay between tapping a grade button and the next card appearing.
3. AI practice → `POST /api/generate` sends due cards to Claude via `lib/generate-practice.ts` to create ephemeral extra exercises (not persisted).

**Knowledge graph (2026-06 refactor):** Every card stores `components` (JSON string[] of base-form Hangul lemmas it is *built from* — the prerequisites). At sync time these are resolved to `CardDependency` edges. At study time, `GET /api/cards/due` fetches the in-session edges and passes them to `sequenceCards()` (`lib/sequence.ts`), which applies a **blended weighted score**: `score = depth − urgencyBoost`, sort ascending. `depth` = longest in-session prereq chain; `urgencyBoost = min(daysOverdue / 7, 3)`. Foundations come first; a card overdue by 3+ weeks can leapfrog up to 3 prerequisite levels. Cycle-safe (visited-stack DFS). Pure, testable.

**Doc emphasis capture:** `lib/google-docs.ts` now reads `textStyle` (bold / underline / backgroundColor) per text run and returns `{ text, emphasized }[]` per lesson. `emphasized` is a deduped list of the explicitly highlighted spans — these are passed to the extraction prompt as a "MUST make a card for each" signal. Body text stays clean (no inline markers) so `targetForm` substring matching is unaffected.

**Exhaustive extraction & dedup:**  `lib/extract-cards.ts` uses `claude-opus-4-8` with `thinking: { type: 'adaptive' }` and streams the response (`.finalMessage()`). Prompt is **exhaustive** — every distinct item taught becomes a card (one card per base form; conjugations go in `notes`). Function words get a card only when the lesson explicitly teaches them; otherwise they appear only in `components`. Each card has a new `components: string[]` field. `lib/card-key.ts` exports `normalizeFront(front)` — strips English glosses in trailing parens, NFC-normalizes, collapses whitespace — used as the DB unique key. Real dedup is enforced by `Card.normalizedFront @unique`; Claude's hint list is a secondary courtesy signal. When editing a card's front, the `[id]/route.ts` update also sets `normalizedFront`.

**Sentence-centric learning:** Cards carry `Sentence[]` — 1–3 natural Korean example sentences, each storing `korean` (full sentence), `targetForm` (exact surface form to highlight/blank), `translation`, and (at session time) `unknownCount` (number of non-target tokens not yet seen by the learner). Sentences are **presentation context only** — `Card` remains the FSRS review unit. `lib/sentence-match.ts` is the single source of truth for substring matching/blanking (handles 1-char/multi-occurrence safety rules). `components/HighlightedSentence.tsx` renders the highlighted span. The **shown sentence** is the one with fewest unknown words (`unknownCount`); ties break via `(hashStr(card.id) + reps) % candidates.length` for variety across reviews. Blank-safety overrides the least-unknown pick when `needsBlank` — falls back to the first blank-safe sentence. The flashcard mode has an **Exposure** (sentence shown, or bare word if context not yet learnable) / **Recall** (word blanked) sub-toggle. **Blank-safety:** the first sentence for every card must be blank-safe (targetForm 2+ characters, appears exactly once) — the prompt guarantees this.

**Tap-to-gloss (2026-06 P2):** Tapping any word in a Korean example sentence shows a popover with its dictionary form, gloss, and part of speech. Resolution order: (1) corpus-first — `normalizeFront(word)` → exact card lookup (instant); (2) stem fallback via `splitParticle`; (3) Setting-table cache (key prefix `gloss:`) for previously-looked-up unknowns; (4) LLM fallback via `claude-haiku-4-5-20251001` (fast, single-word lookups). Cache write is non-blocking. `lib/gloss.ts` owns the prompt + cache helpers. `app/api/gloss/route.ts` is the POST endpoint. `components/GlossProvider.tsx` is a client context mounted once in `app/layout.tsx` — exposes `useWordTap()` hook (returns `showGloss(word, anchorRect)` or `undefined` when not mounted); owns the single popover instance and an in-memory request cache (`useRef<Map>`). "Add as card?" in the popover POSTs to `/api/cards`. `components/HighlightedSentence.tsx` accepts an optional `onWordTap` callback; when provided it renders each word token in a `TappableSegment` (inline `<span role="button">`).

**Audio TTS — swappable provider (2026-06 P2):** `lib/tts.ts` exports a `TtsProvider` interface and `activeTtsProvider` (chosen by `TTS_PROVIDER` env; default `'google'`). Two providers ship: `googleNeural2Provider` (ko-KR-Neural2-A, reuses `GOOGLE_SERVICE_ACCOUNT_KEY` with `cloud-platform` scope — requires the Cloud Text-to-Speech API to be enabled on the GCP project) and `elevenLabsProvider` (`eleven_multilingual_v2`, requires `ELEVENLABS_API_KEY`; currently active). `app/api/tts/route.ts` is a `GET ?text=&voice=` endpoint: hashes `(provider.id, voice, text)` → checks Vercel Blob (`head()`); cache hit → returns stable public URL; miss → synthesizes, `put()` to Blob, returns URL. Reads `KOREAN_BLOB_READ_WRITE_TOKEN` (falls back to `BLOB_READ_WRITE_TOKEN`); returns 503 when unset so the client degrades gracefully. `components/AudioButton.tsx` — three-state speaker button (idle/loading/playing); `haptic('selection')` on press; fetches `/api/tts` → `new Audio(url).play()`; falls back to `window.speechSynthesis` (ko-KR, rate 0.9) on non-200. Wired into `StudySession.tsx` on every sentence and every Korean card front, across all three study modes.

**Cards page — iOS-native (2026-06 P2):** Sticky top bar: single search input + `SlidersHorizontal` filter icon + active-filter count badge. The filter icon opens a `Sheet` (reuses `components/Sheet.tsx`) holding the type pills + `LessonRangeFilter`. Card rows are wrapped in `components/SwipeRow.tsx` (pointer-based horizontal swipe → reveals a red Delete action; reduced-motion safe). `CardEditor` renders inside a `Sheet` keyed by `editingId` instead of in-place. The `'sentences'` segmented view is renamed **"Reading practice"** (`ActiveView = 'cards' | 'reading-practice'`). (This interactivity now lives in `components/CardsClient.tsx` — see the RSC hydration entry below.)

**RSC server hydration + DTO pattern (2026-07 v1.2):** Every main route (`/cards`, `/study`, `/`, `/habits`) follows the same shape: `app/*/page.tsx` is a thin **async RSC** (no `'use client'`, no hooks) that fetches its initial data server-side via Prisma/`lib/` calls and renders exactly one `*Client.tsx` component with the data as props — `CardsClient`, `StudyClient`, `HomeClient`, `HabitsClient`. Real data arrives in the initial server-rendered HTML, so there is never an empty-state or blank-loading flash on first paint; each client shell starts directly in its "ready" state (populated list, `'select-mode'`, populated hero, etc.) instead of a `loading` phase. **DTO contract (RSC-05):** no raw Prisma `Date` object may cross the server→client prop boundary — every `DateTime` field is `.toISOString()`'d and typed as `string` in `lib/dto.ts` (`CardDTO`, `SentenceDTO`, `ReviewDTO`, `LessonDTO`, `LessonRefDTO`, `StatsDTO`, `ActivityDTO`). **Shared pipeline modules:** the data-fetching logic that used to live inline in API routes was extracted into server-only `lib/` functions so the RSC page and the legacy API route call the same code — `lib/study-cards.ts:getStudyCards()` (used by both `app/study/page.tsx` and `GET /api/cards/due`) and `lib/dashboard.ts:getStats()`/`getActivityData()` (used by `app/page.tsx`, `app/habits/page.tsx`, `GET /api/stats`, `GET /api/activity`). **Navigation skeletons:** `app/cards/loading.tsx`, `app/study/loading.tsx`, `app/habits/loading.tsx` are static server components (no `'use client'`, no hooks) using `bg-surface-2 animate-pulse` exclusively — Next.js shows these automatically during client-side route transitions (they do not cover first-load, which is handled by the RSC hydration above).

**Delight — copy voice, icon, My Korean (2026-06 P2):** `lib/copy.ts` — pure warm-copy helpers (`comebackMessage`, `bandUpMessage`, `sessionCompleteMessage`, `hundredDayMessage`, `atRiskMessage`). `components/HabitTracker.tsx` shows a green comeback pill when a freeze bridged the previous missed day. `app/page.tsx` detects a CEFR band-up (compares `computeProficiency().band` to `localStorage('lastBand')`) and shows a confetti + warm banner. App icon: `public/icon.svg` (한 on brand-blue rounded-square); `scripts/gen-icons.mjs` rasterizes it via `sharp` → `icon-192.png`, `icon-512.png`, `apple-icon.png`, `icon-512-maskable.png` (maskable safe-zone variant). `app/manifest.ts` updated (`background_color: #f9fafb`, maskable entry). `app/wrapped/page.tsx` — "My Korean" summary page: fetches `/api/activity` + `/api/stats`, renders streak + CEFR band (`ProficiencyArc`) + all-time stats grid + next-milestone card; share via `navigator.share` / clipboard fallback. Entry points on Home + Habits pages.

**No romanization:** card fronts must be Hangul, never Latin-letter romanization (e.g. `(kkujunhada)`). Short **English clarifying glosses** in parentheses are fine (e.g. `~(으)로 (direction particle)`). `normalizeFront` strips these glosses when computing the dedup key.

**Color system & theming (2026-06 P1 + P2 addition):** Card-type taxonomy is recolored off the primary action color — vocabulary→indigo, grammar→violet, phrase→teal (`lib/card-style.ts:typeBadgeClass` is the single source for type badges; **blue is reserved for actions**). Semantic CSS tokens live in `app/globals.css`: `--surface-1/2/3` (**surface-1** = elevated card/sheet/nav, most prominent in both themes; **surface-2** = quiet recessed strip/tile; **surface-3** = deep well), `--reward` (user-configurable warm accent for streaks/goal/celebration), `--reward-soft` (derived `color-mix` shade for partial-progress tier — bars, heatmap cells, streak dots that are between 0 and goal), `--highlight-bg/fg` (the sentence marker), `--cat-vocab/grammar/phrase`. Tokens are exposed as Tailwind utilities via `@theme inline` (e.g. `bg-surface-1`, `text-cat-vocab`, `bg-reward`, `bg-reward-soft`). Stat/habit displays use these — never literal `orange-*` or `blue-*`. **Two configurable accents:** `buttonColor` (action — buttons/links/nav) and `rewardColor` (reward — streaks/goal/celebration) are both DB settings, injected as `--button`/`--button-foreground` and `--reward`/`--reward-foreground` on `<html>` at server render. `lib/palettes.ts` defines 6 curated complementary pairings (cool action + warm reward) and `findActivePaletteId(action, reward)` for the Settings UI. The Settings **App colors** section lets you pick a pairing (sets both at once) or expand **Customize** to adjust each color independently. **Dark mode is a manual System/Light/Dark toggle:** a pre-paint `<script>` in `app/layout.tsx` resolves the choice (from `localStorage` key `theme`; absent = System) and sets `data-theme` on `<html>` before first paint (no flash; `<html suppressHydrationWarning>`). Tailwind's `dark:` variant is rebound to `[data-theme="dark"]` via `@custom-variant dark` in `globals.css`; the OS `@media (prefers-color-scheme: dark)` block stays as a no-JS fallback for the CSS variables. **When adding a dark value, mirror it in BOTH the media-query `:root` block and the `:root[data-theme="dark"]` block.** `lib/theme.ts` (`getStoredTheme`/`applyTheme`/`resolveTheme`) + `components/ThemeWatcher.tsx` (live OS-change sync) drive the Settings ▸ Appearance control. Theme is **client-only (localStorage), not a DB Setting**; `buttonColor`/`rewardColor` (DB) and `--reading-scale` / `hangul-spaced` (reading aid) are still injected as inline `style`/class on `<html>` server-side in `layout.tsx`.

**Auth:** A single shared-password gate. `middleware.ts` guards all pages/APIs except `/login`, `/api/login`, and static/PWA assets; `lib/auth.ts` issues/verifies an HMAC session cookie (Web Crypto). Configured via `APP_PASSWORD` and `AUTH_SECRET`.

Add new settings here (theme is localStorage; the rest are DB settings via `lib/settings.ts`).

**Database (Prisma + libSQL):**
- `Lesson` → raw doc snapshot per section + contentHash for dedup + **`orderIndex` (Int, 1-based)** for stable lesson numbering. New lessons get `max(orderIndex) + 1` on sync.
- `Card` → front (Korean), back (English), type (vocabulary/grammar/phrase), notes, `distractors` (JSON), **`normalizedFront String @unique`** (dedup key via `normalizeFront()`), **`components String?`** (JSON string[] of prerequisite lemmas). `lessonId` = the lesson where this card was **first introduced** (never overwritten on re-sync). `clozeSentence`/`clozeAnswer`/`clozeTranslation` are **deprecated columns** (kept in DB, no longer written).
- `Sentence` → example sentence attached to a `Card` (CASCADE delete). Fields: `korean`, `targetForm`, `translation`, `orderIndex` (0-based). 1–3 per card.
- **`CardDependency`** *(new 2026-06)* → directed prerequisite edge: `cardId` is built from `prerequisiteId`. `@@unique([cardId, prerequisiteId])`. Created at sync time by resolving `Card.components` lemmas → matching `normalizedFront`. Scripts: `apply-graph-ddl.mjs` (one-time DDL), `relink-dependencies.mjs` (full retroactive relink after a corpus-wide resync).
- `CardReview` → FSRS state per card (stability, difficulty, state, reps, lapses, nextReview)
- `StudyDay` → per-day active study time (`date` = user-local habit-day "YYYY-MM-DD", `seconds`, `reviews`)
- `Setting` → generic key/value app settings: `dailyGoalSeconds`, `habitDayStartHour` (int 0–23, default 2), `sessionSize` (int 5–100, default 20), `readingTextScale` (0.9–1.4), `readingAid` ('0'/'1'), `buttonColor` (hex, action accent), `rewardColor` (hex, reward/streak accent). Also stores tap-to-gloss lookup cache under `gloss:<normalizedWord>` prefix (JSON `GlossResult`; non-fatal if missing). NB: the light/dark **theme** is client-only (localStorage), not stored here.

**Habit tracking:** `components/StudySession.tsx` measures active study time and flushes increments to `POST /api/activity` (keyed by the client's **habit-day** date). Use `habitDateStr(hour)` from `lib/habit.ts` wherever you need "today as a habit date"; never use raw `localDateStr()` for activity logging.

**Lesson filtering:** `GET /api/lessons` returns all lessons ordered by `orderIndex`. Both the Cards page and the Study page share `components/LessonRangeFilter.tsx`. Cards page filters client-side; Study page passes `?lessonFrom=&lessonTo=` to `/api/cards/due` server-side. Unassigned cards (`lessonId = null`) are included only under the full "everything" span.

**Study-ahead:** After clearing due cards, the completion screen offers "Study N more →". This calls `GET /api/cards/due?scope=ahead`, which returns up to `sessionSize` not-yet-due cards nearest to becoming due, also sequenced foundation-first. These are real FSRS reviews.

The app uses the **libSQL adapter** (`lib/prisma.ts`), so the same code runs against a local SQLite file in dev and **hosted Turso** in production.

## Scripts (`scripts/`)

One-time / operational scripts — run locally against Turso via `@libsql/client` or `npx tsx`:

- `apply-graph-ddl.mjs` — Adds `normalizedFront`, `components` columns to `Card` and creates `CardDependency` table. Run ONCE after a clean wipe (UNIQUE index requires an empty table).
- `wipe-card-data.mjs` — Deletes all `CardDependency`, `Sentence`, `CardReview`, `Card`, `Lesson` rows. Keeps `StudyDay` and `Setting`. Safe to re-run.
- `local-resync.mts` — Full re-extraction of all lessons locally (bypasses 60 s Vercel timeout). Run with `npx tsx scripts/local-resync.mts`. Idempotent — skips already-synced lessons by contentHash.
- `relink-dependencies.mjs` — Retroactively rebuilds all `CardDependency` edges from stored `components`. Useful after a full resync to catch cross-lesson forward references. Idempotent.
- `find-duplicates.mjs` — Scans all cards for near-duplicate fronts using a fuzzy key (strips `~`, all parens). Reports any groups of 2+ for manual review.
- `full-resync.mjs` — Drives repeated `POST /api/sync` calls against the live URL until `remaining=0`. Use for incremental syncs; prefer `local-resync.mts` for bulk.
- `apply-sentence-ddl.mjs` — One-time DDL that created the `Sentence` table (already applied, do not re-run).
- `add-lesson-order.mjs` — One-time backfill that added `orderIndex` to existing lessons (already applied, do not re-run).
- `gen-icons.mjs` — Rasterizes `public/icon.svg` (한 mark on brand blue) via `sharp` into the icon set: `icon-192.png`, `icon-512.png`, `apple-icon.png` (180px), `icon-512-maskable.png` (한 scaled to 80% safe zone). Run locally after editing the SVG: `node scripts/gen-icons.mjs`, then commit the PNGs. Requires Apple SD Gothic Neo or another Korean font on the OS.

## Environment Variables

Local dev uses `.env` / `.env.local`. Required keys:
- `ANTHROPIC_API_KEY`
- `GOOGLE_SERVICE_ACCOUNT_KEY` — the full service-account JSON (single line). The Docs API needs OAuth2, not an API key; `lib/google-docs.ts` mints a read-only (`documents.readonly`) access token from this via `google-auth-library`. The target Doc must be readable by the service account (link-view or explicitly shared).
- `NEXT_PUBLIC_GOOGLE_DOC_ID`
- `DATABASE_URL` (libsql://… for Turso; `file:./dev.db` for a purely local DB) and `DATABASE_AUTH_TOKEN` (Turso)
- `APP_PASSWORD` (the login password) and `AUTH_SECRET` (random string for signing the auth cookie)
- `TTS_PROVIDER` — `'google'` or `'elevenlabs'` (default `'google'`; currently set to `'elevenlabs'`)
- `ELEVENLABS_API_KEY` — required when `TTS_PROVIDER=elevenlabs`
- `KOREAN_BLOB_READ_WRITE_TOKEN` — Vercel Blob store token for TTS MP3 caching (must be a **public** store; route also checks `BLOB_READ_WRITE_TOKEN` as fallback). Set in `.env.local` and Vercel env. When missing, `/api/tts` returns 503 and `AudioButton` falls back to browser speech.
- For `TTS_PROVIDER=google`: the Cloud Text-to-Speech API must be enabled on the GCP project that owns `GOOGLE_SERVICE_ACCOUNT_KEY`; the token is minted with `cloud-platform` scope in `lib/tts.ts` (no new credentials needed).

## Gotchas / conventions

- **Vercel function timeout is 60 s hard limit on Hobby plan** — `maxDuration = 300` in the route code has no effect on Hobby. Keep each sync request to 1 lesson. Use `local-resync.mts` for bulk operations.
- **ESLint is strict** (`eslint-config-next` 16). Two rules bite often:
  - `react-hooks/purity` — no impure calls during render (`Date.now()`, `Math.random()`, no-arg `new Date()`). Read time/randomness in effects/event handlers, or via a seeded value (see `seededShuffle` and the `seed` memo in `StudySession.tsx`).
  - `react-hooks/set-state-in-effect` — don't call `setState` synchronously in an effect body; do it inside async callbacks (`fetch().then(setX)`) or handlers.
- **Lint is clean** (`npm run lint` passes with zero errors). Keep new code clean.
- **Git:** commits are on `main` and auto-deploy to Vercel on push to GitHub.
- **`local-resync.mts` env loading:** it uses dynamic `import()` for lib modules so that `dotenv.config()` runs before Prisma/Anthropic read `process.env`. Static `import` statements are hoisted in ESM and would pick up the wrong `DATABASE_URL`.
- **Vercel CLI:** installed globally at `/opt/homebrew/bin/vercel`. If it's ever missing, `npm i -g vercel`. The `APP_PASSWORD` for the login page is in `.env.local`.
- **Keep this file and `.planning/codebase/*.md` current:** refresh both after every milestone close (`.planning/PROJECT.md` § Evolution has the checklist). Verify claims against actual source before writing them — these docs have drifted from reality before (e.g. `.planning/codebase/TESTING.md` claimed zero test coverage after Vitest tests already existed).
- **`loading.tsx` only covers client-side navigation, not first load:** first-load speed on `/cards`, `/study`, `/habits` comes from the RSC hydration pattern (data arrives with the HTML); the `loading.tsx` skeleton is what Next.js shows during a `<Link>` navigation between routes while the destination RSC is fetching. Test RSC first-paint behavior with `npm run build && npm start`, not `next dev`.
- **RSC pages stay data-only, no interactivity:** when adding to `app/*/page.tsx`, keep it a thin async server component (fetch + render one `*Client.tsx`). All state, event handlers, and hooks belong in the `*Client.tsx` shell — this is what keeps the DTO boundary (RSC-05) enforceable and the page itself trivially correct.
