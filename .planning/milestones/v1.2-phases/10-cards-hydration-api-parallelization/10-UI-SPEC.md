# Phase 10 — UI Design Contract

> Visual and interaction contract for Phase 10: Cards Hydration + API Parallelization.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Character

This phase is a **structural refactor, not a visual change.** The user must see no visual difference before and after the conversion. The design contract exists to:

1. Confirm all existing tokens, animations, and interaction patterns are preserved in the new `CardsClient` component
2. Specify the one new visible state: **hydrated-on-first-paint** (no empty-state flash)
3. Lock the loading transition contract so the Phase 9 skeleton → hydrated content handoff is seamless

No new components, colors, typography sizes, or spacing values are introduced.

## Skeleton → Content Transition Contract

The Phase 9 `loading.tsx` skeleton is the navigation-time fallback. The RSC conversion (Phase 10) affects **first-load** behavior. These two states must be visually compatible:

| State | Trigger | Visual |
|-------|---------|--------|
| Skeleton (Phase 9) | Client-side navigation to /cards | `bg-surface-3 animate-pulse` shapes from `loading.tsx` |
| Hydrated (Phase 10) | First load / RSC response complete | Real card list with `bg-surface-1 rounded-xl shadow-sm` rows |
| Empty (pre-Phase 10) | First load before cards fetched | "No cards stored yet" empty state — **must not appear on first load after conversion** |
| Empty (legitimate) | Zero cards in DB | Empty state copy (see Copywriting Contract) |

Transition rules:
- Skeleton shapes in `loading.tsx` approximate real card row height (`p-4 flex flex-col gap-2` with 3–4 lines) — **do not change loading.tsx skeleton in this phase**
- No fade or crossfade animation is needed for the skeleton → hydrated handoff; Next.js replaces the loading UI automatically
- The view-toggle pill (Cards / Reading practice) must be present at hydration time, not added client-side

## Interaction Preservation Contract

All existing interactions in `app/cards/page.tsx` must work identically in the extracted `CardsClient` component. This is the verification checklist, not new design work:

| Interaction | Component | State managed by |
|-------------|-----------|-----------------|
| Search input | top bar `<input>` | `useState<string>` in CardsClient |
| Type filter pills (all / vocabulary / grammar / phrase) | filter Sheet | `useState<string>` in CardsClient |
| Lesson range filter | `LessonRangeFilter` inside Sheet | `useState<number>` pair in CardsClient (initialized from `initialLessons` prop) |
| Filter icon + badge count | `SlidersHorizontal` button | derived from filter + lessonFrom/To state |
| View toggle (Cards / Reading practice) | pill toggle | `useState<ActiveView>` in CardsClient |
| Swipe-to-delete | `SwipeRow` | inline handler → DELETE /api/cards/[id] |
| Tap to edit | card row tap → Sheet | `useState<string | null>` editingId in CardsClient |
| CardEditor save | Sheet → PUT /api/cards/[id] | `handleSave` updates local state |
| Add card sheet | "+" button → Sheet | `useState<boolean>` showAdd in CardsClient |
| Tap-to-gloss | `useWordTap()` hook | `GlossProvider` context (unchanged) |
| Collapse/expand type groups | chevron button per group | `useState<Record<string, boolean>>` in CardsClient |

Post-mutation refresh: `loadCards()` calls `fetch('/api/cards')` after add/edit/delete — this is the only client-side fetch that remains. It replaces the full card list. This is intentional and must be preserved.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | "Add card" (existing — preserve exactly) |
| Empty state heading | "No cards stored yet" (existing — appears only when DB has zero cards after hydration) |

## DTO Boundary — Visual Impact

The DTO serialization (RSC-05) has no visual impact but has one interaction-layer implication:

- `CardReview.nextReview` and `lastReview` arrive as ISO strings in `CardsClient` (not Date objects)
- The cards page currently only uses `review.reps` and `review.lapses` for display (rep count badge) — no date formatting occurs in the cards list view
- If any future display of dates is added to `CardsClient`, use `new Date(isoString).toLocaleDateString()` — never pass the raw ISO string to a date-formatting utility expecting a `Date` object
