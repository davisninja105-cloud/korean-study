# Phase 12 ŌĆö UI Design Contract

> Visual and interaction contract for the Home & Habits Hydration phase.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase intent (from 12-CONTEXT.md):** Convert `app/page.tsx` and `app/habits/page.tsx` from client-side `useEffect` fetches to RSC + client-shell + DTO. **No new visual design, copy, or components are introduced.** This contract exists to lock the existing visual surface as invariant across the serverŌåÆclient boundary so the executor does not regress first-paint UX, copy voice, or the one-tick skeleton/pulse behavior during the conversion.

**Preserve-mode rule:** Where this spec says "preserve," the executor must not alter classnames, copy strings, token usage, or DOM shape from the current `app/page.tsx` / `app/habits/page.tsx` / `app/habits/loading.tsx`. The RSC conversion moves data acquisition server-side; the rendered output must be byte-equivalent (modulo the one-tick skeleton/pulse swap per D-02).

- **Home hero:** No empty-state flash is permitted on first paint. `HomeClient` receives non-null `initialStats` + `initialActivity` from the RSC. **Error state contract:**
- **Home RSC fetch failure:** The home RSC follows the Phase 11 precedent ŌĆö no `try/catch` at the RSC page level; Next.js `error.tsx` / the existing error boundary handles a `lib/dashboard.ts` throw. (Consistent with the codebase convention "No try/catch at RSC page level" from STATE.md.)
- **`lib/dashboard.ts` failure:** `getStats()` / `getActivityData()` wrap `Promise.all` in try/catch per codebase convention; a transient Prisma failure throws and surfaces through the RSC error boundary, not as an empty hero.
- **Home post-sync refetch failure:** Client-side `loadStats()` / `loadActivity()` already use `.catch(() => {})` (non-blocking). **Destructive actions in this phase:** None. The band-up banner dismiss (`Ō£Ģ`) is non-destructive (clears a celebration toast; the underlying `localStorage['lastBand']` is already written). No confirmation needed. No card deletes, no settings mutations, no data destruction occurs on the home or habits surface in this phase.

## Phase-12-Specific Interaction Contracts

These are the interaction invariants the RSC conversion must not break. Each maps to a success criterion in `12-CONTEXT.md` / `ROADMAP.md`.

### 1. One-tick skeleton swap (D-02) ŌĆö Habits
- `HabitsClient` initializes `today=''`.
- While `today === ''`, render `<HabitsLoading />` (the **exact** component from `app/habits/loading.tsx` ŌĆö do not invent a new skeleton).
- Compute `today = habitDateStr(dayStartHour)` in a `useEffect` using the established `Promise.resolve().then(() => setToday(...))` pattern (satisfies `react-hooks/set-state-in-effect`).
- Once `today` is set, render the full habits layout with all `useMemo` derivations operating on the RSC-provided `initialDays` / `initialGoal` / `initialMasteredCount`.
- **Visual contract:** the user sees the skeleton for exactly one effect tick, then the real streak/heatmap/trend. No empty-heatmap flash. No new skeleton shapes.

### 2. One-tick hero pulse (D-02) ŌĆö Home
- `HomeClient` initializes `heroState='loading'`.
- While `heroState === 'loading'`, render `<div className="h-28 animate-pulse bg-surface-2 rounded-xl" />` inside the hero section (preserve the exact existing block from `app/page.tsx:174`).
- Derive `heroState` in a `useEffect` from `initialStats` + `initialActivity` + client-local `today` (same logic as current `app/page.tsx:90-103`, operating on props instead of fetched state). Use `Promise.resolve().then(() => setHeroState('A'|'B'|'C'))`.
- **Visual contract:** the user sees the pulse for exactly one effect tick, then the real A/B/C hero. No empty hero flash. Greeting may resolve on the same tick (it does not gate the hero).

### 3. Pull-to-refresh on home (success criteria #3)
- `usePullToRefresh(handleSync)` stays mounted in `HomeClient`. Behavior unchanged: engage only at scroll-top, with resistance; `pullDistance` + `refreshing` drive the indicator.
- `handleSync` calls `POST /api/sync`, then `loadStats()` + `loadActivity()` (client-side refetch via `/api/stats` + `/api/activity` ŌĆö these routes stay per D-08).
- **Visual contract:** indicator copy transitions `Pull to sync` ŌåÆ `Release to sync` ŌåÆ `SyncingŌĆ”` ŌåÆ sync result. The hero may re-pulse briefly if `heroState` is re-derived from the refetched data ŌĆö this is acceptable and matches current behavior.

### 4. Band-up confetti/banner (success criteria #4)
- `HomeClient` compares `computeProficiency(initialStats.masteredCount).band` to `localStorage['lastBand']` on first render-derived effect.
- On mismatch (and `stored` is non-null), show the band-up banner (`bandUpMessage(band, masteredCount)`) and lazy-import `canvas-confetti` (`import('canvas-confetti').then((m) => m.default({ particleCount: 80, spread: 60, origin: { y: 0.4 } }))`).
- Always write `localStorage['lastBand'] = band` after the check.
- **Visual contract:** the banner renders inside `bg-surface-1 rounded-2xl shadow-md px-5 py-4` with `­¤Äē` glyph + dismiss `Ō£Ģ` button. Confetti fires once. After pull-to-refresh, the band-up check re-runs against the refetched `masteredCount`.

### 5. HabitTracker `initialActivity` prop (D-09)
- `HabitTracker` gains optional props `initialDays?`, `initialToday?`, `initialGoal?`.
- When all three are present, skip the `useEffect` self-fetch and seed state from props. When absent, fall back to the existing self-fetch (defensive ŌĆö component stays usable in isolation).
- All existing `HabitTracker` behavior is preserved: milestone detection, haptic on ring close, comeback pill (`comebackMessage` from `lib/copy.ts`), at-risk nudge (`atRiskMessage`).
- **Visual contract:** no visible change to `HabitTracker` rendering. The only observable difference is the absence of a duplicate `/api/activity` request on the home route.

### 6. DTO boundary (RSC-05 ŌĆö success criteria #5)
- `StatsDTO` and `ActivityDTO` are added to `lib/dto.ts` (D-06). `ActivityDTO.days[]` element type is `DayRecord` from `lib/habit.ts` (already serializable: `{ date: "YYYY-MM-DD", seconds: number, reviews?: number }`).
- **No raw `Date` object crosses the serverŌåÆclient boundary** on either route. Stats and activity payloads are already primitives, but the DTO types are formal requirements at the prop boundary.
- The RSC pages (`app/page.tsx`, `app/habits/page.tsx`) are `async` server components that fetch via `lib/dashboard.ts` and serialize to DTO props before rendering `<HomeClient initialStats={...} initialActivity={...} />` / `<HabitsClient initialDays={...} initialGoal={...} initialDayStartHour={...} initialMasteredCount={...} />`.
