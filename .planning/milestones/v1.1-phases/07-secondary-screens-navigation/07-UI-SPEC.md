## Interaction Contract

### Navigation — bottom tab bar (F-04, F-01, SEC-02)

**Safe-area-inset fix (NAV-01 / F-01):**
The live `env(safe-area-inset-bottom)` reference on the nav `pb` and `<main>` `pb` must be replaced with a one-time JS-read CSS variable. Implementation:

```
// In Nav.tsx — useEffect, runs once on mount
useEffect(() => {
  const sab = getComputedStyle(document.documentElement)
    .getPropertyValue('--sab').trim()
  // If not yet set, read from env() which works correctly before first navigation
  if (!sab) {
    const tmp = document.createElement('div')
    tmp.style.paddingBottom = 'env(safe-area-inset-bottom)'
    document.body.appendChild(tmp)
    const value = getComputedStyle(tmp).paddingBottom
    document.body.removeChild(tmp)
    document.documentElement.style.setProperty('--sab', value || '0px')
  }
}, [])
```

Tailwind class replacements:
- Nav `pb-[env(safe-area-inset-bottom)]` → `pb-[var(--sab,0px)]`
- Layout `<main>` `pb-[calc(4.5rem+env(safe-area-inset-bottom))]` → `pb-[calc(4.5rem+var(--sab,0px))]`

**Active/press state (F-04 — SEC-02):**
Each bottom tab `<Link>` currently has no `active:` CSS variant. Add `active:opacity-70` to each tab link's className. This gives immediate visual confirmation of the tap on mobile without changing the layout.

The Settings gear `<Link>` in the top bar must also receive `active:opacity-70`.

**Touch target (SEC-02):**
Bottom tab links use `flex-1 flex flex-col items-center gap-1 py-2`. With `text-xs` label + `w-5 h-5` icon + `gap-1` + `py-2` (16px total padding), the computed height is approximately: 20px icon + 4px gap + 16px label + 16px vertical padding = ~56px. Touch target is already ≥44px. No height change needed — add `min-h-[44px]` as an explicit defensive constraint.

**Token fix:** `bg-gray-100 dark:bg-gray-800` wrapper → `bg-surface-3`; active tab `bg-white dark:bg-gray-700` → `bg-surface-1` (as above in Color section).

## Safe-Area Implementation Detail (NAV-01)

The documented Next.js bug causes `env(safe-area-inset-bottom)` to resolve to `0px` after a client-side `<Link>` navigation on iOS. This happens because the environment variable is re-evaluated after route transitions and returns 0 in non-standalone PWA mode in certain Next.js versions.

**Canonical fix pattern:**

1. In `Nav.tsx` (client component, already `'use client'`), add a `useEffect` that runs once on mount:
   - Creates a temporary element, applies `padding-bottom: env(safe-area-inset-bottom)`, reads the computed value via `getComputedStyle`, removes the element.
   - Writes the result to `document.documentElement.style.setProperty('--sab', value || '0px')`.
2. Replace all live `env(safe-area-inset-bottom)` references with `var(--sab, 0px)`:
   - `components/Nav.tsx` line 64 bottom nav `pb-[env(safe-area-inset-bottom)]` → `pb-[var(--sab,0px)]`
   - `app/layout.tsx` line 74 `<main>` `pb-[calc(4.5rem+env(safe-area-inset-bottom))]` → `pb-[calc(4.5rem+var(--sab,0px))]`

**Why not `useWindowSize` or polling:** The value is stable after the first read. It only breaks when Next.js re-evaluates `env()` on navigation. Reading once at mount captures the correct value while it is still accessible and stores it as a persistent CSS variable.

**Fallback `0px`:** On non-iPhone devices or in desktop browsers, `env(safe-area-inset-bottom)` returns `0px`. The `var(--sab, 0px)` fallback ensures no layout shift when the JS has not yet set the variable (first paint; before `useEffect` fires the `env()` reference in the original class applies, so there is no regression on first paint).
