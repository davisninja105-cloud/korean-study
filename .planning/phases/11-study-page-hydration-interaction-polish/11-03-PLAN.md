<objective>
Eliminate the grade-button jitter (UX-01) by making `submitReview` in `components/StudySession.tsx` optimistic: compute the FSRS result client-side with `reviewCard()`, advance the queue immediately, and persist with a background fire-and-forget `POST /api/review`.

Purpose: Today `submitReview` is `async` and `await fetch('/api/review')` blocks for ~100–200ms before `setQueue` runs — the card visibly stalls between tap and advance. Per D-08 the fix moves the FSRS computation client-side (it is pure and already used client-side via `previewIntervalLabels`), advances the queue right away, and fires the save in the background. Per D-09 a failed background save is silently ignored. Per D-10 undo still works because the snapshot is captured before the advance. This plan is fully independent of the RSC conversion (Plans 01/02) — it touches only `components/StudySession.tsx`.

Output: a synchronous, optimistic `submitReview` and updated call sites.
</objective>

## Interface context

`reviewCard` (`lib/fsrs.ts`, existing export): `reviewCard(review: CardReviewFields, rating: Grade)` where `CardReviewFields` requires `nextReview: Date` and `lastReview: Date | null` (actual Date objects, NOT strings). It returns `{ state, stability, difficulty, elapsedDays, scheduledDays, reps, lapses, nextReview: Date, lastReview: Date }`. It is pure and browser-safe (the same module's `previewIntervalLabels` already runs client-side in `handleReveal`).

`Card.review` shape in `StudySession.tsx` (existing): an object with OPTIONAL numeric fields and `nextReview?: string | null` / `lastReview?: string | null` (ISO strings from the API). These strings must be converted with `new Date(...)` before calling `reviewCard` (Pitfall 1 — string vs Date type mismatch; TypeScript strict mode will reject a string where Date is required).

`nextHabitDayStart(hour, now)` (`lib/habit.ts`, already imported in StudySession): returns the Date at which the current habit day ends; a card is re-queued when its next review falls before that boundary.

`dayStartHourRef` (already in StudySession): `useRef` holding the user's habit day-start hour, loaded by an existing settings effect.
</context_for_executor>
