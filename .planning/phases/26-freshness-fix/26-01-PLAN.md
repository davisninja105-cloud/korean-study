<design_decisions>
This plan set resolves the four open questions carried in STATE.md's Blockers/Concerns. These resolutions bind Plans 26-02 and 26-03:

1. **Mechanism attribution (resolved empirically + by design).** Task 1 re-runs all specs on pristine code and records the current fetch-count signature per red cell. Regardless of what it finds, the fix architecture covers BOTH root-cause mechanisms: `FreshnessWatcher` fires `router.refresh()` at popstate/resume boundaries (fixes 0-fetch Router-Cache reuse), and per-shell gated prop-adoption makes shells adopt fresh props whenever any RSC fetch lands (fixes client-shell non-resync). Neither half alone is sufficient; the plan does not depend on which mechanism each cell currently presents.

2. **/habits post-mutation-return (resolved empirically).** 25-03-SUMMARY re-verified this cell genuinely FRESH; its test already asserts fresh behavior and passes. Task 1 confirms it still passes at baseline. NO fix effort targets this cell and its test is left byte-identical in Task 2.

3. **StudyClient double-fetch risk (resolved by design, implemented in 26-03).** `FreshnessWatcher` fires only on popstate/visibility-resume; `loadDue()` fires only on user filter-Apply. Disjoint triggers — no systematic double-fetch. The only race window (a refresh in-flight when the user taps Apply) resolves safely: `loadDue`'s response wins for the filtered view, and the refresh's server props are discarded by StudyClient's full-span adoption gate. No visibility or popstate listener may be added inside any shell — the watcher is the single boundary-refresh owner.

4. **Per-shell prop-sync gating (resolved by design, implemented in 26-02/26-03).**
   - `HomeClient`: UNGATED adoption of `initialStats`/`initialActivity` on prop-reference change. No clobberable in-flight interaction exists; pull-to-refresh's own `loadStats`/`loadActivity` fetches deliver equivalent server truth (last-writer-wins is fresh either way).
   - `HabitsClient`: remove the read-only `useState` copies of `days`/`goal`/`masteredCount` and read props directly — the state copy IS the bug, and this display-only page has nothing to protect. `today` stays client state.
   - `StudyClient`: adopt `initialCards` only when `phase === 'select-mode' && !isFilterLoading && isFullSpan(lessonFrom, lessonTo, maxOrder)`. Gated payloads are consumed-and-DISCARDED: mid-session the queue owns the cards (FRESH-02), every path back to select-mode already re-fetches (`startAhead`) or crosses a navigation boundary, and server props are always the unfiltered full-span pool so adopting them under a narrowed filter would silently widen the user's chosen range.
   - `CardsClient`: adopt `initialCards` only when `editingId === null && !showAdd && !adding && deletingIds.size === 0`. Gated payloads are consumed-and-DISCARDED, never deferred: a payload produced before a local save/add/delete completed would, if adopted after the sheet closes, overwrite the optimistic local merge with pre-mutation data — the exact FRESH-02 clobber the gate exists to prevent.

Constraints binding all plans: `app/*/page.tsx` and `next.config.ts` are untouched (server-side freshness is already correct via force-dynamic; cache-time tuning is explicitly out of scope per REQUIREMENTS.md). The 7 confirmed-fresh cells and the 4 smoke first-paint assertions must stay green at every step (FRESH-06 / 24-DIAGNOSIS.md Confirmed-Fresh Paths, D-09 of Phase 24's context).
</design_decisions>
