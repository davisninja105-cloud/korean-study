<objective>
Make sync failures diagnosable by naming the specific failed lesson (via a content excerpt the user can search for in their Google Doc) without leaking internal error text, and stop the CardDependency resolver from re-querying the entire deck on every lesson by building the `normalizedFront → cardId` map once per request.

Purpose: Success Criteria 1 (SYNC-01) and 2 (PERF-01) of Phase 14 — the SyncPanel names which lesson failed instead of a meaningless "Lesson 1"/generic remaining count, and the full-deck lookup runs once per request instead of once per lesson.
Output: A hardened, behavior-preserving `app/api/sync/route.ts` — a `lessonExcerpt()` helper feeding sanitized `failures[]`, and a hoisted, incrementally-augmented `keyToId` map for two-phase dependency linking.
</objective>

    Hoist the dependency-resolution map out of the per-lesson loop so the full-deck query runs exactly once per request.

    Before the `for (let i = 0; i < extractResults.length; i++)` loop (i.e. after `orderBase` is read, alongside `newCards`/`newLessons`/`created`/`failures`), declare a shared `const keyToId = new Map<string, string>()` and populate it once from a single `prisma.card.findMany({ select: { id: true, normalizedFront: true }, where: { components: { not: null } } })` — set `keyToId.set(c.normalizedFront, c.id)` for each row. Keep the `where: { components: { not: null } }` filter to preserve the existing edge-creation semantics exactly (only cards that have components are resolvable as prerequisites — do NOT broaden this to all cards; that would change which edges are created and is out of scope for a performance-preserving refactor). Note the `select` no longer needs `components` because component strings now come from the in-memory extraction result, not a DB re-read.

    Inside the per-lesson upsert `Promise.all` over `cards`, collect a per-lesson `const linkTargets: { id: string; components: string[] }[] = []` (declare it inside the loop body so it resets each lesson). In the CREATE branch, after `prisma.card.create` resolves, when `card.components.length > 0` do both `keyToId.set(nf, created.id)` and `linkTargets.push({ id: created.id, components: card.components })`. In the UPDATE branch, when `card.components.length > 0` do both `keyToId.set(nf, existing.id)` and `linkTargets.push({ id: existing.id, components: card.components })`. Because the linking step runs only after `await Promise.all(...)` resolves, `keyToId` is fully populated with this lesson's cards by link time (forward references within the lesson resolve, matching prior behavior).

    Replace the linking block (lines 187-232): remove the per-lesson `prisma.card.findMany` + local `keyToId` construction + the `upsertedIds`/`upsertedSet` bookkeeping. Instead iterate `linkTargets`; for each `{ id, components }`, for each `comp` (skip falsy), compute `prereqId = keyToId.get(normalizeFront(comp))`, skip when `!prereqId || prereqId === id`, and `prisma.cardDependency.upsert(...)` with the same `cardId_prerequisiteId` compound key, `create: { cardId: id, prerequisiteId: prereqId }`, `update: {}` no-op — each wrapped in the existing per-edge try/catch that `console.warn`s and continues (link failures stay non-fatal). Drop the now-unused `upsertedIds` array and its `.push` calls; keep `createdThisLesson` and `newCards += createdThisLesson` unchanged.

    Do NOT change: the `existingNormalizedFronts` query at lines 51-53 (a different concern — the Claude dedup hint list), the extraction `Promise.allSettled`, the Lesson create / orphan-cleanup try/catch, or the final response shape. This task is a pure structural/performance change with no user-visible behavior difference.
