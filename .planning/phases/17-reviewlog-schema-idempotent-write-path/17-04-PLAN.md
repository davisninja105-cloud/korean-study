<objective>
Close the single blocker gap from Phase 17 end-of-phase UAT (Test 6): a retried/duplicate POST /api/review with the same idempotencyKey returns 500 instead of the intended idempotent 200 replay.

Root cause (fully diagnosed in `.planning/debug/reviewlog-p2002-catch-never-fires.md`): the route's write path uses an INTERACTIVE `prisma.$transaction(async (tx) => {...})` callback (required by CR-01's optimistic-concurrency read-then-conditional-write). With Prisma 7 + `@prisma/adapter-libsql`, a SQLite UNIQUE-constraint violation thrown inside that callback surfaces as a raw, unclassified `DriverAdapterError` (`cause.kind: 'sqlite'`, message contains `UNIQUE constraint failed: ReviewLog.idempotencyKey`) rather than a `Prisma.PrismaClientKnownRequestError` with `code: 'P2002'`. The catch block only tests `instanceof Prisma.PrismaClientKnownRequestError`, so it never matches, and every duplicate falls through to the generic 500.

Fix direction (the lower-risk option (a) recommended by the debug session — preserves the interactive transaction and its CR-01 optimistic-concurrency logic): widen the idempotent-200 catch branch to ALSO recognize the raw DriverAdapterError shape for this specific constraint, keeping the existing P2002 check for forward-compatibility.

To make the detection logic itself verifiable (not just "it compiles"), the raw-error match is extracted into a pure, unit-testable helper. This is load-bearing, not scope creep: requirement #2 of this gap is that the fix must NOT swallow unrelated failures as false success — a property that is proven cheaply by unit-testing the negative cases and cannot be proven by a single manual curl.

Purpose: Restore HIST-02 (idempotent write path) — the core guarantee this phase exists to deliver.
Output: A pure `lib/db-errors.ts` helper + its unit test, and a widened catch condition in `app/api/review/route.ts`. Zero new npm dependencies. `app/api/review/undo/route.ts` untouched. No other Phase 17 deliverable re-planned.
</objective>
