# Phase 17 — Security

> Per-phase security contract: threat register, accepted risks, and audit trail.

---

## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| local script → Turso DB | `scripts/apply-reviewlog-ddl.mjs` applies structural DDL to the production database using credentials from `.env` | DDL statements; `DATABASE_URL`/`DATABASE_AUTH_TOKEN` |
| client → POST /api/review | Untrusted request body (`cardId`, `rating`, `idempotencyKey`) crosses into the transactional FSRS write path | Card grading input |
| API → Turso DB | The route relies on the DB UNIQUE constraint on `ReviewLog.idempotencyKey` to enforce exactly-once semantics | FSRS state + audit row writes |
| user undo action → in-flight background retry | An undo must cancel a not-yet-delivered retry so a stale rating cannot re-apply after the review is reverted | `AbortController` cancel signal |

## Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation | Status |
|-----------|----------|-----------|----------|-------------|------------|--------|
| T-17-01 (schema layer) | Tampering | `ReviewLog.idempotencyKey` UNIQUE index | high | mitigate | DB-level UNIQUE index (`prisma/schema.prisma:106`, applied via `scripts/apply-reviewlog-ddl.mjs`) is the source of truth that makes dedup enforceable. Verified: `idempotencyKey String @unique` present; DDL script creates `ReviewLog_idempotencyKey_key` UNIQUE INDEX. | closed |
| T-17-01 (API layer) | Tampering | `POST /api/review` lost-response retry double-applying FSRS state | high | mitigate | Idempotency key + DB UNIQUE constraint + atomic `prisma.$transaction`; P2002 on `idempotencyKey` caught and returned as idempotent 200 reading back current state instead of re-running `reviewCard()`. Verified in `app/api/review/route.ts:79-146` (transaction wraps read/update/create; P2002 branch checks `e.meta?.target` includes `idempotencyKey` per WR-01 fix). | closed |
| T-17-01 (client layer) | Tampering | Per-attempt key generation defeating server-side dedup | high | mitigate | Key generated exactly once in `submitReview` (event handler, not render) and passed unchanged through every retry attempt. Verified: `components/StudySession.tsx:557` (`crypto.randomUUID()`) called once per grade; same `idempotencyKey` threaded into `postReviewWithRetry` and reused across all 3 attempts. | closed |
| T-17-02 | Tampering | Stale background retry re-applying a rating after undo | high | mitigate | Per-grade `AbortController` on single-slot `undoRef.current`; `handleUndo` calls `controller.abort()` before issuing the undo request; `postReviewWithRetry` checks `cancelSignal.aborted` at loop-top, in the catch block, and after each backoff wait. Verified in `components/StudySession.tsx:112-164` and `617-624`. Best-effort for an already-in-flight request at the instant of abort (accepted narrow residual window, documented in 17-03-PLAN.md); covers the dominant multi-second backoff window. | closed |
| T-17-03 | Denial of Service | Overly long / malformed `idempotencyKey` → unbounded storage growth | medium | mitigate | Input validation rejects non-string / empty / length > 100 with 400 before any DB call. Verified in `app/api/review/route.ts:58-63`. | closed |
| T-17-06 | Tampering | Impure key/controller generation breaking render (`react-hooks/purity`) | low | mitigate | `crypto.randomUUID()` / `new AbortController()` stay in the event-handler body (`submitReview`, called only from click/keyboard handlers — verified `components/StudySession.tsx:466,614,709-716`), never render/`useMemo`. `npm run lint` passes with 0 errors. | closed |
| T-13-02 | Information Disclosure | Raw Prisma/Turso error leaking to client | medium | mitigate | Full error logged server-side via `console.error`; client receives generic `{ error: 'Failed to record review' }` 500. Verified in `app/api/review/route.ts:147-151`. | closed |
| T-17-04 | Denial of Service | Idempotency-key guessing/forgery to suppress a legitimate review write | low | accept | Single shared-password app behind the existing HMAC cookie gate (`middleware.ts`); the key is generated by the same trusted client that would send the real request — no third-party-control threat model. | closed |
| T-17-05 | Tampering | DDL applied to production Turso | low | accept | Operation is non-destructive (`CREATE TABLE IF NOT EXISTS` on a net-new table) and idempotent; no existing table/data is altered. Standard manual-DDL workflow per `CLAUDE.md`. | closed |
| T-17-SC | Tampering | npm/pip/cargo installs (supply chain) | n/a | accept | No package installs across all three plans in this phase — uses only already-installed `@libsql/client`, `@prisma/client`, and platform built-ins (`crypto.randomUUID`, `AbortController`). Supply-chain vector N/A. | closed |

**Note (bonus finding, non-blocking):** the post-plan code review (`17-REVIEW.md` / `17-REVIEW-FIX.md`) surfaced and fixed an additional Tampering-adjacent issue not in the original threat model — **CR-01**: a non-atomic read-modify-write in `POST /api/review` that allowed a lost FSRS update between two *distinct* concurrent reviews of the same card (separate from the idempotency-retry threat above, which covers the *same* request being retried). Fixed via an optimistic-concurrency guard (`updateMany` scoped to a `{reps, lastReview}` version fingerprint, `count === 0` → 409) inside the same transaction. Recorded here for traceability; not part of the L1 register short-circuit above since it was discovered and closed before this audit ran.
