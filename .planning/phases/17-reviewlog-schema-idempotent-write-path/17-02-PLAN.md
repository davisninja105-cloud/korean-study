<objective>
Harden `POST /api/review` into an idempotent, transactional write path: validate the client-supplied idempotency key, write the CardReview update and the ReviewLog audit row inside one atomic array-form transaction, and treat a duplicate-key collision as an idempotent success.

Purpose: Delivers HIST-01 (every review writes a ReviewLog row) and the server half of HIST-02 (a lost-response retry never duplicates a row or double-applies FSRS state). The DB UNIQUE constraint from 17-01 is the enforcement mechanism; this route composes it with the existing FSRS write.
Output: A modified `app/api/review/route.ts` that accepts `idempotencyKey`, runs `prisma.$transaction([cardReview.update, reviewLog.create])`, and catches P2002 as a 200 idempotent replay.
</objective>

4. Inside the existing `try`, keep the `findUnique` + `if (!cardReview) return 404` + `const updated = reviewCard(cardReview, rating)` lines exactly as they are. Replace the single `prisma.cardReview.update({ where: { cardId }, data: updated })` call with an array-form transaction: `const [review] = await prisma.$transaction([ prisma.cardReview.update({ where: { cardId }, data: updated }), prisma.reviewLog.create({ data: { cardId, rating, idempotencyKey, ...updated } }) ])` then `return NextResponse.json(review)`. `updated` is the resulting post-update FSRS object (D-02) — it holds only the FSRS scalar fields (state/stability/difficulty/elapsedDays/scheduledDays/reps/lapses/nextReview/lastReview), so spreading it alongside cardId/rating/idempotencyKey into reviewLog.create is correct and complete. Do NOT use the callback form `$transaction(async (tx) => ...)` — array form is the only form proven safe with the libSQL adapter here.

5. In the existing `catch (e)` block, BEFORE the current generic `console.error` + 500, add a branch: `if (e instanceof Prisma.PrismaClientKnownRequestError && e.code === 'P2002')`. On that branch, re-fetch `const current = await prisma.cardReview.findUnique({ where: { cardId } })` and `return NextResponse.json(current)` (implicit 200). This is the idempotent replay — the whole array transaction rolled back so CardReview did NOT re-apply, meaning the current stored state is already correct. Do NOT inspect `e.meta.target` (Pitfall 3 — idempotencyKey is the only reachable unique constraint in this transaction; meta.target is unreliably populated on libSQL anyway).

6. Preserve the generic 500 fallthrough with server-only `console.error('POST /api/review failed:', e)` and never leak the raw error to the client (T-13-02 disclosure posture, unchanged).

Do NOT touch `app/api/review/undo/route.ts` — undo never writes or mutates ReviewLog (HIST-06). Run `npx prisma generate` first if `prisma.reviewLog` types are not present in this working tree (17-01 generated them; `npm run build` also regenerates via prebuild).
