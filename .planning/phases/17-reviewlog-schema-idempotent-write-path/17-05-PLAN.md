<objective>
Close the single remaining blocker from `17-VERIFICATION.md`: a duplicate-idempotencyKey POST to /api/review still returns 500 instead of an idempotent 200 for a SECOND, distinct error shape the 17-04 fix does not cover.

The gap (verbatim from verification): the collision is classified by Prisma as `PrismaClientKnownRequestError`/P2002, but `e.meta.target` is `undefined` (only `meta.modelName` and `meta.driverAdapterError` are populated), and the informative constraint text lives at `e.meta.driverAdapterError.cause.originalMessage` (e.g. `"SQLITE_CONSTRAINT: UNIQUE constraint failed: ReviewLog.idempotencyKey"`). The current `isUniqueConstraintError()` only walks `error` → `error.cause` and only inspects a `message` field — it never descends into `.meta.driverAdapterError` nor reads `originalMessage`, so it misses this shape. The route's own `e.meta?.target?.includes('idempotencyKey')` sub-condition also never fires (target is undefined). Net: the request falls through to the generic 500.

This exact requirement (HIST-02's core idempotent-replay guarantee) has now been "verified by hand, then regressed" twice — UAT Test 6, then this retroactive verification — because there has never been a persisted automated regression test protecting it. This plan fixes the detection blind spot AND adds that missing test.

Fix insight (minimal blast radius): the route's catch already ORs in `isUniqueConstraintError(e, 'idempotencyKey')`. Once the helper recognizes the new shape, the existing branch returns 200 automatically — so `app/api/review/route.ts` needs NO change. The whole behavioral fix is a generalization of the pure helper.

Purpose: Make HIST-02 reliably true across BOTH real error-classification shapes this Prisma 7 + `@prisma/adapter-libsql` stack produces for the identical underlying SQLite UNIQUE-constraint violation.
Output: An extended pure `lib/db-errors.ts` helper (+ new unit cases), a new persisted `tests/review-route.test.ts` regression test (real handler + local SQLite file DB), and a minimal `vitest.config.ts` that lets tests import the route through the `@/` alias. Zero new dependencies. `app/api/review/route.ts` and `app/api/review/undo/route.ts` untouched. Existing `tests/db-errors.test.ts` (10/10) and the full 126-test suite stay green.
</objective>
