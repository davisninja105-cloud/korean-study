## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → API (`/api/review`, `/api/cards/[id]`) | Untrusted JSON body (`cardId`, `rating`, `front`, `sentences`) crosses here; requests are authenticated by the single shared-password HMAC cookie (middleware.ts) but body contents are attacker-controllable. |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-13-01 | Tampering | `POST /api/review` `rating` field | medium | mitigate | REVIEW-02 range guard rejects any rating outside integer {1,2,3,4} with a 400 before `reviewCard()` runs, preventing undefined FSRS behavior / throws from crafted input. |
| T-13-02 | Information Disclosure | `POST /api/review` 500 path | low | mitigate | REVIEW-01 catch logs the raw error server-side (`console.error`) and returns a generic `{ error: 'Failed to record review' }` — internal Prisma/schema details never reach the client. |
| T-13-03 | Information Disclosure | `PUT /api/cards/[id]` collision 400 message | low | accept | The friendly 400 confirms a front already exists, theoretically enabling front-enumeration — but this is a single-tenant app (one shared password, no multi-tenancy per .claude/CLAUDE.md Architectural Constraints), so the only party who can enumerate is the sole owner of their own cards. Accepted; message wording is required by ROADMAP Success Criterion 3. |
| T-13-04 | Denial of Service | `POST /api/review` unhandled throw | low | mitigate | REVIEW-01 try/catch prevents a DB hiccup from throwing an unhandled 500 and destabilizing the route; failures now return a controlled response. |
| T-13-SC | Tampering | dependency installs | n/a | accept | No package-manager installs in this plan — the P2002 error class ships with the already-generated Prisma client; no supply-chain surface added. |
