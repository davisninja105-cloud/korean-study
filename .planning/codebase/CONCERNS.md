# Codebase Concerns

**Analysis Date:** 2026-07-10

Verified against source post-v1.5 close (Phase 23). All 241 Vitest tests pass (18 files). Concerns resolved since the prior audit are noted at the bottom.

## Tech Debt

**Gloss cache soft cap with no eviction:**
- Issue: `setCachedGloss` in `lib/gloss.ts` enforces `MAX_GLOSS_CACHE_ENTRIES = 2000` (lines 40–49) by silently returning when the count is at cap — no LRU eviction, no TTL. Once full, new glosses are never cached and old entries are never removed.
- Files: `lib/gloss.ts`, `app/api/gloss/preload/route.ts`
- Impact: Cache hit rate degrades permanently once the cap is reached; every new unknown word costs a Haiku call forever. `Setting`-table prefix scans for `gloss:` keys are O(n).
- Fix approach: Add a `createdAt`/last-access timestamp to the cached JSON and evict oldest on insert, or move the cache to a dedicated table with an index and TTL cleanup.

**Prisma error classification workaround (libSQL adapter):**
- Issue: UNIQUE-constraint violations inside interactive `prisma.$transaction(async tx => …)` surface as raw `DriverAdapterError` (`cause.kind: 'sqlite'`), not classified `P2002`. `lib/db-errors.ts` implements a breadth-first traversal of nested error fields as a workaround; `app/api/review/route.ts` checks both P2002 and `isUniqueConstraintError`.
- Files: `lib/db-errors.ts`, `app/api/review/route.ts`
- Impact: If Prisma/adapter error wrapping changes on upgrade, idempotent-retry detection could silently stop working, turning benign retried reviews into 500s. Any new route using interactive transactions inherits the risk.
- Fix approach: `tests/db-errors.test.ts` covers the current shapes — keep it updated on every Prisma upgrade. Prefer array-form transactions where possible (they classify correctly).

**StudySession.tsx still large despite mode extraction:**
- Issue: The three study modes were extracted (`components/FlashcardMode.tsx`, `MultipleChoiceMode.tsx`, `FillBlankMode.tsx`), but `components/StudySession.tsx` remains 832 lines owning queue management, FSRS grading, retry chains, undo, activity flushing, and sentence selection wiring.
- Files: `components/StudySession.tsx`
- Impact: High cognitive load; the undo/retry/idempotency interplay (undoRef, controller, isMountedRef guards) is subtle and easy to break during unrelated edits.
- Fix approach: Extract the review-save/undo machinery (`postReviewWithRetry`, undo snapshot lifecycle) into a testable hook or service module. The pure parts already have tests via `tests/review-route.test.ts`.

**Deprecated cloze columns retained in schema:**
- Issue: `Card.clozeSentence`/`clozeAnswer`/`clozeTranslation` are deprecated (no longer written) but remain in `prisma/schema.prisma` and the generated client.
- Files: `prisma/schema.prisma`
- Impact: Schema noise; new code could accidentally read stale values.
- Fix approach: Drop via manual libSQL DDL (Turso migration constraint) at a convenient milestone boundary.

**Generated Prisma client committed under `app/generated/`:**
- Issue: ~13,000 lines of generated Prisma model code live inside `app/generated/prisma/` (e.g. `models/Card.ts` at 2,503 lines), inflating diffs and file-size scans.
- Files: `app/generated/prisma/`
- Impact: Cosmetic/tooling — grep and wc results are dominated by generated code; risk of accidental hand-edits.
- Fix approach: Ensure lint/search tooling excludes the directory; document that it is regenerated by `npx prisma generate`.

**Extraction truncation-salvage path depends on SDK internals:**
- Issue: `lib/extract-cards.ts` now uses native structured outputs (`output_config.format` + `zodOutputFormat`), but the truncation fallback registers a `stream.on('text')` accumulator and salvages raw text through `parseExtractionResponse` when `finalMessage()` **rejects** with an `AnthropicError` — behavior that relies on the SDK's format-parser throwing inside the stream's `message_stop` handler.
- Files: `lib/extract-cards.ts` (531 lines)
- Impact: An `@anthropic-ai/sdk` upgrade that changes how parse failures surface (e.g., resolving with `parsed_output: null` instead of rejecting) would silently disable salvage, dropping truncated lessons' cards.
- Fix approach: `tests/extract-cards.test.ts` covers the salvage path — re-verify against the SDK on every upgrade; also handle a resolved-with-null `parsed_output` defensively.

## Known Bugs

**Sentence-match particle-splitting ambiguity:**
- Symptoms: `splitParticle` in `lib/sentence-match.ts` can mis-split multi-syllable verb stems with modifier endings (e.g. 기다리는 → 기다리 + 는 vs. whole word) — an orthographic ambiguity with no morphological analyzer.
- Files: `lib/sentence-match.ts`, `components/HighlightedSentence.tsx`
- Trigger: Grammar cards whose targetForm ends in a single-char case marker attached to a long stem.
- Workaround: The split is deliberately conservative (single-char markers require a 2+ syllable stem; 도/만/나 excluded). Mis-splits affect highlight tinting only, not FSRS correctness.

## Security Considerations

**Single shared-password auth with 1-year non-expiring session:**
- Risk: One `APP_PASSWORD` gates everything; the HMAC cookie (`lib/auth.ts`) is set with `maxAge: 60*60*24*365` (`app/api/login/route.ts`) and there is no rotation/invalidation short of changing `AUTH_SECRET`.
- Files: `lib/auth.ts`, `middleware.ts`, `app/api/login/route.ts`
- Current mitigation: Stateless HMAC (forgery requires `AUTH_SECRET`); middleware fails closed if `AUTH_SECRET` is missing; cron path fails closed too (`isValidCronAuth` rejects when `CRON_SECRET` is unset).
- Recommendations: Add a token version/timestamp to the HMAC message so sessions can be invalidated; consider a shorter maxAge.

**No rate limiting on any API route:**
- Risk: `/api/login` (password guessing), `/api/sync` and `/api/gloss` (Anthropic token burn), `/api/tts` (ElevenLabs + Blob cost) have no per-IP throttling. Login is the only unauthenticated one, but authenticated abuse of the LLM routes is also unmetered.
- Files: `middleware.ts`, `app/api/login/route.ts`, `app/api/gloss/route.ts`, `app/api/sync/route.ts`, `app/api/tts/route.ts`
- Current mitigation: Single-tenant, password-gated; sync processes only 1 lesson per call.
- Recommendations: Add a small sliding-window limiter in edge middleware for `/api/login` at minimum; consider daily budget caps on the LLM routes.

**Secrets present but properly externalized:**
- Risk: Low. `.env` / `.env.local` exist locally (not committed); all credentials are read from env (`ANTHROPIC_API_KEY`, `GOOGLE_SERVICE_ACCOUNT_KEY`, `DATABASE_AUTH_TOKEN`, `ELEVENLABS_API_KEY`, `APP_PASSWORD`, `AUTH_SECRET`, `CRON_SECRET`, Blob token).
- Current mitigation: No hardcoded secrets found in source.
- Recommendations: None urgent.

## Performance Bottlenecks

**One lesson per sync (Vercel Hobby 60 s hard limit):**
- Problem: `MAX_LESSONS_PER_SYNC = 1` (`lib/sync.ts`); Opus exhaustive extraction of one lesson can take 30–90 s, so a 20-lesson backlog needs 20 sync taps (or the daily cron).
- Files: `lib/sync.ts`, `app/api/sync/route.ts`, `app/api/cron/sync/route.ts`
- Cause: Hobby plan ignores `maxDuration`; extraction is LLM-bound.
- Improvement path: `scripts/local-resync.mts` for bulk (documented, working). Cron drains the backlog one lesson/day. Time-budgeted multi-lesson batching would help but adds complexity.

**Auto-relink adds a whole-graph pass to every successful new-lesson sync:**
- Problem: `runSync` (`lib/sync.ts` line ~333) calls `relinkAllDependencies()` after each sync that adds a lesson — 2 full reads (all cards + all edges) plus ≤1 `createMany`, inside the same 60 s budget as extraction.
- Files: `lib/sync.ts`, `lib/relink-dependencies.ts`
- Cause: Batch-diff design (RELIABILITY-02/03) trades per-sync cost for correctness of forward-reference edges.
- Improvement path: Cheap at hundreds of cards; if the deck grows to many thousands, scope the relink to the new lesson's lemmas. Relink failure is caught and non-fatal (line ~338).

**Full-deck lookup map built per sync request:**
- Problem: Each sync builds a `normalizedFront → cardId` map over the whole deck for dedup hints and dependency linking.
- Files: `lib/sync.ts`
- Cause: Two-phase linking needs global resolution.
- Improvement path: Fine at current deck size (hundreds of cards); revisit only if the deck grows to many thousands.

**RSC pages have no fetch error fallback:**
- Problem: `app/page.tsx`, `app/study/page.tsx`, `app/cards/page.tsx`, `app/habits/page.tsx` await Prisma-backed lib calls with no try/catch and no `error.tsx` boundaries — a transient DB failure yields the default Next.js error screen.
- Files: `app/*/page.tsx`
- Cause: RSC hydration pattern assumes queries succeed.
- Improvement path: Add `app/error.tsx` (and per-route boundaries where useful) with a retry affordance.

**Google Docs fetch has no retry/backoff:**
- Problem: `lib/google-docs.ts` fetches the doc once with no handling for 403/429/5xx; a rate-limit or transient failure fails the whole sync.
- Files: `lib/google-docs.ts`
- Cause: Simple single-shot fetch.
- Improvement path: Exponential backoff on 429/5xx; the daily cron makes transient failures self-healing, so priority is low.

## Fragile Areas

**LLM response format dependency:**
- Files: `lib/extract-cards.ts`, `lib/gloss.ts`
- Why fragile: Extraction is now schema-valid by construction via structured outputs (`zodOutputFormat(ExtractionSchema)`), but two constraints remain delicate: (1) the zod schema must stay permissive — adding `.max()`/`.min(1)` string constraints would make the whole response throw client-side; (2) the truncation-salvage path depends on SDK rejection behavior (see Tech Debt). Gloss (`lib/gloss.ts`) still regex-extracts JSON from a Haiku text response; parse failures bubble to the caller with no retry.
- Safe modification: Any schema change must keep constraints description-only and update `tests/extract-cards.test.ts` in the same commit.
- Test coverage: Good for parsing/salvage/normalization (241-test suite green); no test exercises live-model schema drift, by nature.

**Review save / undo / idempotency machinery:**
- Files: `components/StudySession.tsx` (queue/undo/retry core), `app/api/review/route.ts`, `app/api/review/undo/route.ts`
- Why fragile: Correctness depends on a chain of invariants — one idempotency key + one AbortController per grade (HIST-02), `handleUndo` aborting the retry chain before the undo POST (HIST-03), `isMountedRef` guards making restores atomic, and the server-side P2002-on-`idempotencyKey`-only retry-success rule. Refactors that "simplify" any link can reintroduce the previously-fixed undo/retry race.
- Safe modification: Preserve the HIST-02/HIST-03 comments and their ordering; extend `tests/review-route.test.ts` when touching the route.
- Test coverage: Route-level tests exist; no client-side race simulation test.

**Dark-mode token duplication:**
- Files: `app/globals.css`
- Why fragile: Every dark token must be mirrored in BOTH the `@media (prefers-color-scheme: dark)` `:root` block and the `:root[data-theme="dark"]` block; a miss produces theme-dependent inconsistency only visible in one of the two paths.
- Safe modification: Grep for the token name in both blocks before committing; no automated consistency check exists.

**Turso schema-change procedure:**
- Files: `prisma/schema.prisma`, `scripts/apply-*-ddl.mjs`
- Why fragile: `prisma db push`/`migrate` do not work against `libsql://`; schema changes require hand-generated DDL applied via `@libsql/client`. Drift between `schema.prisma` and the live Turso DB is possible if a DDL step is skipped.
- Safe modification: Follow the CLAUDE.md 3-step procedure exactly; verify with `turso db shell korean-study`.

## Scaling Limits

**Study pool capped at 1000 due cards:**
- Current capacity: `take: 1000` in `lib/study-cards.ts`.
- Limit: >1000 simultaneously-due cards would be partially invisible per session. Unrealistic for single-tenant usage; harmless today.
- Scaling path: Raise the cap or page the query if the deck grows past ~2–3k cards.

**Gloss cache: 2000-entry hard stop:**
- Current capacity: 2000 `gloss:*` Setting rows; then caching stops entirely (see Tech Debt).
- Scaling path: Eviction/TTL, or dedicated table.

**Prerequisite graph DFS + auto-relink:**
- Current capacity: `lib/sequence.ts` memoized, cycle-safe DFS; `relinkAllDependencies` is a whole-corpus batch-diff. Both fine at hundreds of cards/edges.
- Limit: None practical at this scale.
- Scaling path: Scope relink to new-lesson lemmas if the deck grows to many thousands.

## Dependencies at Risk

**Claude API (Opus extraction + Haiku gloss):**
- Risk: Model deprecation (`claude-opus-4-8`, `claude-haiku-4-5-20251001` pinned) or outage blocks sync and degrades tap-to-gloss. Structured outputs (`output_config.format`) also couples extraction to a specific SDK/API feature.
- Impact: Sync fails loudly (per-lesson error surfaced in SyncPanel); gloss falls back to corpus/cache tiers.
- Migration plan: Model IDs are single constants in `lib/extract-cards.ts` / `lib/gloss.ts` — one-line swap; re-verify salvage tests and structured-output behavior after any model/SDK change.

**Google Docs API:**
- Risk: Outage or credential expiry (`GOOGLE_SERVICE_ACCOUNT_KEY`) blocks new-lesson ingestion only.
- Impact: Existing study data unaffected; daily cron retries naturally.
- Migration plan: None needed; content lives in the tutor's doc by design.

**Vercel Blob (TTS cache) + ElevenLabs:**
- Risk: Missing/invalid `KOREAN_BLOB_READ_WRITE_TOKEN` → `/api/tts` returns 503; ElevenLabs outage → synthesis fails.
- Impact: `AudioButton` falls back to `window.speechSynthesis` (robotic but functional); no server-side alerting when this happens.
- Migration plan: `lib/tts.ts` provider abstraction — flip `TTS_PROVIDER=google` (Neural2) as the fallback provider.

## Missing Critical Features

**No error boundaries on RSC routes:**
- Problem: No `app/error.tsx` or per-route error components; any server fetch failure shows the framework error page.
- Blocks: Graceful recovery from transient DB failures without a manual reload.

**No observability channel for degradations:**
- Problem: TTS 503 fallback, gloss-cache-full, and gloss parse failures still degrade with no alerting channel; known-lemmas degradation is now logged (`lib/study-cards.ts` line 81, `console.error`) but only to Vercel function logs — nothing surfaces to the user or a dashboard.
- Blocks: Detecting quality regressions (robotic audio, cold gloss cache) before the user notices.

- Files: `app/api/**/route.ts`

**Client-side undo/retry race:**

- Files: `components/StudySession.tsx`
- Risk: A refactor could drop the `controller.abort()` before-undo ordering without test failure.

**Google Docs parsing:**

- Priority: Medium (failures are loud at sync time, but fixture-based tests would localize them).

## Resolved Since Prior Audit (2026-07-06)

- **Known-lemmas query degrades silently** — fixed (Phase 23-01): `lib/study-cards.ts` now logs the rejection via `console.error` before degrading to an empty Set (test-covered in `tests/study-cards.test.ts`). Remaining gap is observability-channel only (logs, no alerting).
- **Forward-reference CardDependency edges silently skipped** — fixed (Phase 23-02, RELIABILITY-02/03): `runSync` auto-runs `relinkAllDependencies()` (`lib/relink-dependencies.ts` → `lib/link-dependencies.ts:computeMissingEdges`) after every successful sync that adds a lesson; idempotent, non-fatal on failure. `scripts/relink-dependencies.mts` remains as a manual fallback.
- **Regex-based extraction parsing as primary path** — improved (Phase 22, PROMPT-01): extraction now uses native structured outputs; `parseExtractionResponse` is demoted to a truncation-salvage fallback (new SDK-coupling concern noted in Tech Debt).
- **Corpus quality drift** — addressed (Phase 22): `lib/audit-checks.ts` + `scripts/retro-filter-cleanup.mts` audit/fix persisted cards; fixes applied and verified.
