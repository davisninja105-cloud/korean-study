# Codebase Concerns

**Analysis Date:** 2026-07-06

Verified against source post-v1.4 close. All 130 Vitest tests pass (14 files). Several concerns from prior audits are now resolved and noted at the bottom.

## Tech Debt

**Gloss cache soft cap with no eviction:**
- Issue: `setCachedGloss` in `lib/gloss.ts` (lines 40‚Äď49) enforces `MAX_GLOSS_CACHE_ENTRIES = 2000` by silently returning when the count is at cap ‚ÄĒ no LRU eviction, no TTL. Once full, new glosses are never cached and old entries are never removed.
- Files: `lib/gloss.ts`, `app/api/gloss/preload/route.ts`
- Impact: Cache hit rate degrades permanently once the cap is reached; every new unknown word costs a Haiku call forever. `Setting`-table prefix scans for `gloss:` keys are O(n).
- Fix approach: Add a `createdAt`/last-access timestamp to the cached JSON and evict oldest on insert, or move the cache to a dedicated table with an index and TTL cleanup.

**Prisma error classification workaround (libSQL adapter):**
- Issue: UNIQUE-constraint violations inside interactive `prisma.$transaction(async tx => ‚Ä¶)` surface as raw `DriverAdapterError` (`cause.kind: 'sqlite'`), not classified `P2002`. `lib/db-errors.ts` implements a breadth-first traversal of nested error fields as a workaround; `app/api/review/route.ts` checks both P2002 and `isUniqueConstraintError` (lines 120‚Äď126).
- Files: `lib/db-errors.ts`, `app/api/review/route.ts`
- Impact: If Prisma/adapter error wrapping changes on upgrade, idempotent-retry detection could silently stop working, turning benign retried reviews into 500s. Any new route using interactive transactions inherits the risk.
- Fix approach: `tests/db-errors.test.ts` covers the current shapes ‚ÄĒ keep it updated on every Prisma upgrade. Prefer array-form transactions where possible (they classify correctly).

**StudySession.tsx still large despite mode extraction:**
- Issue: The three study modes were extracted (`components/FlashcardMode.tsx`, `MultipleChoiceMode.tsx`, `FillBlankMode.tsx`), but `components/StudySession.tsx` remains 832 lines owning queue management, FSRS grading, retry chains, undo, activity flushing, and sentence selection wiring.
- Files: `components/StudySession.tsx`
- Impact: High cognitive load; the undo/retry/idempotency interplay (undoRef, controller, isMountedRef guards) is subtle and easy to break during unrelated edits.
- Fix approach: Extract the review-save/undo machinery (`postReviewWithRetry`, undo snapshot lifecycle) into a testable hook or service module. The pure parts already have tests via `tests/review-route.test.ts`.

**Deprecated cloze columns retained in schema:**
- Issue: `Card.clozeSentence`/`clozeAnswer`/`clozeTranslation` are deprecated (no longer written) but remain in `prisma/schema.prisma` and the generated client.
- Files: `prisma/schema.prisma`
- Impact: Schema noise; new code could accidentally read stale values.
- Fix approach: Drop via manual libSQL DDL (Turso migration constraint) at a convenient milestone boundary.

**Generated Prisma client committed under `app/generated/`:**
- Issue: ~13,000 lines of generated Prisma model code live inside `app/generated/prisma/` (e.g. `models/Card.ts` at 2,503 lines), inflating diffs and file-size scans.
- Files: `app/generated/prisma/`
- Impact: Cosmetic/tooling ‚ÄĒ grep and wc results are dominated by generated code; risk of accidental hand-edits.
- Fix approach: Ensure lint/search tooling excludes the directory; document that it is regenerated by `npx prisma generate`.

## Known Bugs

**Sentence-match particle-splitting ambiguity:**
- Symptoms: `splitParticle` in `lib/sentence-match.ts` can mis-split multi-syllable verb stems with modifier endings (e.g. ÍłįŽč§Ž¶¨ŽäĒ ‚Üí ÍłįŽč§Ž¶¨ + ŽäĒ vs. whole word) ‚ÄĒ an orthographic ambiguity with no morphological analyzer.
- Files: `lib/sentence-match.ts`, `components/HighlightedSentence.tsx`
- Trigger: Grammar cards whose targetForm ends in a single-char case marker attached to a long stem.
- Workaround: The split is deliberately conservative (single-char markers require a 2+ syllable stem; ŽŹĄ/ŽßĆ/Žāė excluded). Mis-splits affect highlight tinting only, not FSRS correctness.

**Known-lemmas query degrades silently:**
- Symptoms: `lib/study-cards.ts` fetches the due pool and known-lemmas set concurrently via `Promise.allSettled` (lines 36‚Äď39); a known-lemmas failure degrades to an empty Set with no logging.
- Files: `lib/study-cards.ts`
- Trigger: Transient Turso failure on the second query.
- Workaround: Session still works ‚ÄĒ sentence ranking just loses the "prefer known context" signal; new-card bare-word gating may over-trigger. Consider logging the rejection so degradation is visible.

**Forward-reference CardDependency edges silently skipped:**
- Symptoms: At sync time, `components[]` lemmas that name a card not yet in the deck (taught in a later lesson) create no edge; the graph is silently incomplete until `scripts/relink-dependencies.mjs` is run manually.
- Files: `lib/sync.ts`, `lib/link-dependencies.ts`
- Trigger: Incremental one-lesson-at-a-time syncs of a doc with cross-lesson forward references.
- Workaround: Run `relink-dependencies.mjs` after a full corpus sync. Could be automated as a final pass once `remaining=0`.

## Security Considerations

**Single shared-password auth with 1-year non-expiring session:**
- Risk: One `APP_PASSWORD` gates everything; the HMAC cookie (`lib/auth.ts`) is set with `maxAge: 60*60*24*365` (`app/api/login/route.ts` line 21) and there is no rotation/invalidation short of changing `AUTH_SECRET`.
- Files: `lib/auth.ts`, `middleware.ts`, `app/api/login/route.ts`
- Current mitigation: Stateless HMAC (forgery requires `AUTH_SECRET`); middleware fails closed if `AUTH_SECRET` is missing; cron path fails closed too (`isValidCronAuth` rejects when `CRON_SECRET` is unset ‚ÄĒ `lib/auth.ts` lines 30‚Äď33).
- Recommendations: Add a token version/timestamp to the HMAC message so sessions can be invalidated; consider a shorter maxAge.

**No rate limiting on any API route:**
- Risk: `/api/login` (password guessing), `/api/sync` and `/api/gloss` (Anthropic token burn), `/api/tts` (ElevenLabs + Blob cost) have no per-IP throttling. Login is the only unauthenticated one, but authenticated abuse of the LLM routes is also unmetered.
- Files: `middleware.ts`, `app/api/login/route.ts`, `app/api/gloss/route.ts`, `app/api/sync/route.ts`, `app/api/tts/route.ts`
- Current mitigation: Single-tenant, password-gated; sync processes only 1 lesson per call.
- Recommendations: Add a small sliding-window limiter in edge middleware for `/api/login` at minimum; consider daily budget caps on the LLM routes.

**Secrets present but properly externalized:**
- Risk: Low. `.env` / `.env.local` exist locally (not committed); all credentials are read from env (`ANTHROPIC_API_KEY`, `GOOGLE_SERVICE_ACCOUNT_KEY`, `DATABASE_AUTH_TOKEN`, `ELEVENLABS_API_KEY`, `APP_PASSWORD`, `AUTH_SECRET`, `CRON_SECRET`, Blob token).
- Current mitigation: No hardcoded secrets found in source.
- Recommendations: None urgent.

## Performance Bottlenecks

**One lesson per sync (Vercel Hobby 60 s hard limit):**
- Problem: `MAX_LESSONS_PER_SYNC = 1` (`lib/sync.ts` line 19); Opus exhaustive extraction of one lesson can take 30‚Äď90 s, so a 20-lesson backlog needs 20 sync taps (or the daily cron).
- Files: `lib/sync.ts`, `app/api/sync/route.ts`, `app/api/cron/sync/route.ts`
- Cause: Hobby plan ignores `maxDuration`; extraction is LLM-bound.
- Improvement path: `scripts/local-resync.mts` for bulk (documented, working). Cron drains the backlog one lesson/day. Time-budgeted multi-lesson batching would help but adds complexity.

**Full-deck lookup map built per sync request:**
- Problem: Each sync builds a `normalizedFront ‚Üí cardId` map over the whole deck for dedup hints and dependency linking.
- Files: `lib/sync.ts`
- Cause: Two-phase linking needs global resolution.
- Improvement path: Fine at current deck size (hundreds of cards); revisit only if the deck grows to many thousands.

**RSC pages have no fetch error fallback:**
- Problem: `app/page.tsx`, `app/study/page.tsx`, `app/cards/page.tsx`, `app/habits/page.tsx` await Prisma-backed lib calls with no try/catch and no `error.tsx` boundaries ‚ÄĒ a transient DB failure yields the default Next.js error screen.
- Files: `app/*/page.tsx`
- Cause: RSC hydration pattern assumes queries succeed.
- Improvement path: Add `app/error.tsx` (and per-route boundaries where useful) with a retry affordance.

**Google Docs fetch has no retry/backoff:**
- Problem: `lib/google-docs.ts` fetches the doc once with no handling for 403/429/5xx; a rate-limit or transient failure fails the whole sync (grep confirms no retry logic).
- Files: `lib/google-docs.ts`
- Cause: Simple single-shot fetch.
- Improvement path: Exponential backoff on 429/5xx; the daily cron makes transient failures self-healing, so priority is low.

## Fragile Areas

**LLM response format dependency:**
- Files: `lib/extract-cards.ts`, `lib/gloss.ts`
- Why fragile: Both assume specific JSON shapes from Claude. Extraction has an `isValidExtractedCard` guard plus truncation-salvage parsing (`parseExtractionResponse`, WR-01) ‚ÄĒ now covered by `tests/extract-cards.test.ts` including mid-nested-array truncation ‚ÄĒ but format drift can still silently drop cards. Gloss parse failures bubble to the caller with no retry.
- Safe modification: Any prompt-schema change must update the salvage tests in `tests/extract-cards.test.ts` in the same commit.
- Test coverage: Good for parsing (`extract-cards.test.ts`, 130-test suite green); no test exercises a live-model schema drift, by nature.

**Review save / undo / idempotency machinery:**
- Files: `components/StudySession.tsx` (lines ~100‚Äď260, 550‚Äď650), `app/api/review/route.ts`, `app/api/review/undo/route.ts`
- Why fragile: Correctness depends on a chain of invariants ‚ÄĒ one idempotency key + one AbortController per grade (HIST-02), `handleUndo` aborting the retry chain before the undo POST (HIST-03), `isMountedRef` guards making restores atomic, and the server-side P2002-on-`idempotencyKey`-only retry-success rule (WR-01). Refactors that "simplify" any link can reintroduce the previously-fixed undo/retry race.
- Safe modification: Preserve the HIST-02/HIST-03 comments and their ordering; extend `tests/review-route.test.ts` when touching the route.
- Test coverage: Route-level tests exist; no client-side race simulation test.

**Dark-mode token duplication:**
- Files: `app/globals.css`
- Why fragile: Every dark token must be mirrored in BOTH the `@media (prefers-color-scheme: dark)` `:root` block and the `:root[data-theme="dark"]` block; a miss produces theme-dependent inconsistency only visible in one of the two paths.
- Safe modification: Grep for the token name in both blocks before committing; no automated consistency check exists.

**Turso schema-change procedure:**
- Files: `prisma/schema.prisma`, `scripts/apply-*-ddl.mjs`
- Why fragile: `prisma db push`/`migrate` do not work against `libsql://`; schema changes require hand-generated DDL applied via `@libsql/client`. Drift between `schema.prisma` and the live Turso DB is possible if a DDL step is skipped.
- Safe modification: Follow the CLAUDE.md 3-step procedure exactly; verify with `turso db shell korean-study`.

## Scaling Limits

**Study pool capped at 1000 due cards:**
- Current capacity: `take: 1000` in `lib/study-cards.ts` (line 56).
- Limit: >1000 simultaneously-due cards would be partially invisible per session. Unrealistic for single-tenant usage; harmless today.
- Scaling path: Raise the cap or page the query if the deck grows past ~2‚Äď3k cards.

**Gloss cache: 2000-entry hard stop:**
- Current capacity: 2000 `gloss:*` Setting rows; then caching stops entirely (see Tech Debt).
- Scaling path: Eviction/TTL, or dedicated table.

**Prerequisite graph DFS:**
- Current capacity: `lib/sequence.ts` memoized, cycle-safe DFS; fine at hundreds of cards/edges.
- Limit: None practical at this scale.
- Scaling path: N/A for the foreseeable deck size.

## Dependencies at Risk

**Claude API (Opus extraction + Haiku gloss):**
- Risk: Model deprecation (`claude-opus-4-8`, `claude-haiku-4-5-20251001` pinned) or outage blocks sync and degrades tap-to-gloss.
- Impact: Sync fails loudly (per-lesson error surfaced in SyncPanel); gloss falls back to corpus/cache tiers.
- Migration plan: Model IDs are single constants in `lib/extract-cards.ts` / `lib/gloss.ts` ‚ÄĒ one-line swap; re-verify extraction JSON schema against salvage tests after any model change.

**Google Docs API:**
- Risk: Outage or credential expiry (`GOOGLE_SERVICE_ACCOUNT_KEY`) blocks new-lesson ingestion only.
- Impact: Existing study data unaffected; daily cron retries naturally.
- Migration plan: None needed; content lives in the tutor's doc by design.

**Vercel Blob (TTS cache) + ElevenLabs:**
- Risk: Missing/invalid `KOREAN_BLOB_READ_WRITE_TOKEN` ‚Üí `/api/tts` returns 503; ElevenLabs outage ‚Üí synthesis fails.
- Impact: `AudioButton` falls back to `window.speechSynthesis` (robotic but functional); no server-side alerting when this happens.
- Migration plan: `lib/tts.ts` provider abstraction ‚ÄĒ flip `TTS_PROVIDER=google` (Neural2) as the fallback provider.

## Missing Critical Features

**No error boundaries on RSC routes:**
- Problem: No `app/error.tsx` or per-route error components; any server fetch failure shows the framework error page.
- Blocks: Graceful recovery from transient DB failures without a manual reload.

**No observability for silent degradations:**
- Problem: Known-lemmas fallback, TTS 503 fallback, gloss-cache-full, and skipped forward-reference edges all degrade silently with no logging/alerting channel.
- Blocks: Detecting quality regressions (worse sentence ranking, robotic audio, cold gloss cache) before the user notices.
