## Directory Purposes

### `app/`

Next.js 16 App Router root. Subdirectories with `page.tsx` are routes (React Server Components by default). Subdirectories under `api/` with `route.ts` are serverless API handlers.

**RSC Pattern (v1.2):** The four main routes (`/`, `/study`, `/cards`, `/habits`) follow a consistent pattern:
1. Async server component (no `'use client'`)
2. Fetch data via Prisma or `lib/` functions
3. Serialize all `Date` fields to ISO strings
4. Render exactly one `*Client.tsx` component from `components/` with data as props

This eliminates empty-state loading flashes on first paint. The page itself is trivially correct (just data + render).

**Navigation Fallbacks:** `loading.tsx` siblings in `/study`, `/cards`, `/habits` provide Next.js-managed skeletons for client-side route transitions. These are shown only when the user clicks a link; first-load skips them (RSC hydration).

### `app/api/`

Thin request handlers. Each route follows the same pattern:
1. Extract + validate input (return 400 on bad schema)
2. Call `lib/` functions or Prisma
3. Serialize response (convert `Date` → ISO strings)
4. Return JSON; catch errors and return 500

**No business logic lives in API routes.** Complex logic is extracted to `lib/` so both RSC pages and API routes share it:
- `lib/study-cards.ts:getStudyCards()` — used by `app/study/page.tsx` (RSC) and `GET /api/cards/due`
- `lib/dashboard.ts:getStats()` — used by `app/page.tsx` (RSC) and `GET /api/stats`

### `app/generated/prisma/`

Auto-generated Prisma client. Created by `npx prisma generate` from `prisma/schema.prisma`. Never edited by hand. The `output` directive in `schema.prisma` points here so the generated client works in the Vercel build environment.

### `components/`

All shared React UI in a flat directory (no sub-grouping by domain). Components that need browser APIs or interactivity declare `'use client'` as the first line. Purely presentational components may be RSC-compatible.

**Client Shells (v1.2):** The top-level interactive components for each route:
- `HomeClient.tsx` — receives `StatsDTO` + `ActivityDTO` props
- `StudyClient.tsx` — receives `CardDTO[]` + `LessonDTO[]` props
- `CardsClient.tsx` — receives `CardDTO[]` + `LessonDTO[]` props
- `HabitsClient.tsx` — receives `ActivityDTO` + `masteredCount` props

Client shells manage all state/hooks and render the rest of the UI. **No data-fetching in `useEffect`** — data arrives as RSC props.

**Import Rules:** No component imports from `app/`. Components import from `@/lib` (for pure logic) and `@/components` (for other UI).

### `lib/`

Business logic, utilities, and server-side helpers (30+ files). All files are importable from both API routes and RSC pages. **No imports from `app/` or `components/`.**

**Module Categories:**

### `prisma/`

Contains only `schema.prisma`. No migration files (Turso doesn't support `prisma migrate`). The only Prisma commands run in CI/build are:
- `npx prisma generate` (regenerate client)
- Manual DDL scripts via `@libsql/client` for schema changes

### `scripts/`

Operational scripts for one-time migrations and bulk operations that cannot run inside Vercel's 60-second serverless timeout. Run locally with `node` or `npx tsx`.

**Categories:**

- **Bulk Operations (local only):**
  - `local-resync.mts` — Full corpus re-extraction (preferred for bulk syncs)

  - `wipe-card-data.mjs` — Nuke all card/lesson/review data (keeps StudyDay + Setting)

  - `apply-sentence-ddl.mjs` — Created Sentence table
  - `apply-reviewlog-ddl.mjs` — Created ReviewLog table

  - `backfill-sentences.mjs` — Migrated cloze → Sentence rows

- **Debug/Audit:**
  - `check-edges.mjs` — Validate CardDependency edges

### `tests/`

Vitest unit tests. One file per pure `lib/` module (`card-key`, `habit`, `known-words`, etc.). Scope is deliberately narrow — pure functions with no DB/API dependency.

**Run:**
```bash

npm run test:watch     # Watch mode
```

Tests live alongside source (same directory), not in a separate `__tests__/` folder.

### `public/`

Static files served at `/`. Contains PWA icon set (rasterized from `public/icon.svg` via `scripts/gen-icons.mjs`) and self-hosted fonts.

### `plans/`

Markdown design documents for past/current feature work. Not imported at runtime. Reference only.

### `.planning/`

GSD planning artifacts (created by `/gsd-*` commands). Organized by artifact type: phases, milestones, debug notes, research, UI reviews, etc.

## Naming Conventions

### Files

- **kebab-case** for all `.ts`, `.tsx`, `.mjs`, `.mts` files
  - Examples: `card-key.ts`, `study-cards.ts`, `gloss.ts`
  - **Exception:** React components use PascalCase filenames matching the default export
    - Example: `Sheet.tsx` exports `export default function Sheet()`; `AudioButton.tsx` exports `export default function AudioButton()`

- **Directories:** kebab-case throughout (`app/`, `components/`, `lib/`, `scripts/`, etc.)

### Components

- PascalCase filenames matching the default export (e.g., `Sheet`, `AudioButton`, `StudySession`)
- `'use client'` directive as the very first line if interactive

### API Routes

- Next.js App Router convention: `app/api/<resource>/route.ts`
- Nested resources: `app/api/<resource>/[id]/route.ts`
- Examples: `app/api/cards/route.ts`, `app/api/cards/[id]/route.ts`, `app/api/cards/due/route.ts`

### Variables & Functions

- **camelCase** for all identifiers
- **PascalCase** for React components and type/interface names
- **UPPER_CASE** for constants (e.g., `MAX_LESSONS_PER_SYNC`, `URGENCY_SCALE`)

### Imports

- **Path alias `@/`** for all intra-project imports
  - Examples: `import { prisma } from '@/lib/prisma'`, `import { Sheet } from '@/components/Sheet'`
- **Relative imports** (`./ ../`) only within the same directory
- **No barrel `index.ts` files** — every import names the exact source file

## Where to Add New Code

### New Feature Endpoint

1. **API route:** `app/api/<feature>/route.ts`
   - Validate input (400 on bad schema before business logic)
   - Call `lib/<feature>.ts` functions or Prisma
   - Serialize response (convert `Date` → ISO string)
   - Return JSON; catch errors and return 500

2. **Server-side helper:** `lib/<feature>.ts` (if complex logic)
   - Pure, testable, side-effect-free
   - No imports from `app/` or `components/`
   - Export named functions

3. **Database schema:** Add model to `prisma/schema.prisma`
   - Run `npx prisma generate` to regenerate client
   - For production: extract DDL via `npx prisma migrate diff`, apply manually to Turso

4. **Tests:** `tests/<feature>.test.ts` (if pure lib functions)
   - Vitest syntax
   - Focus on business logic, not UI

### New UI Component

1. **Component:** `components/<FeatureName>.tsx`
   - PascalCase filename
   - Declare `'use client'` as first line if interactive
   - Export as default function

2. **Call from:** A page (`app/*/page.tsx`) or another client component

3. **Imports:** From `@/lib` (pure logic), `@/components` (other UI), never from `app/`

### New Page/Route

1. **Page:** `app/<route>/page.tsx` (async RSC, no `'use client'`)
   - Fetch data server-side via Prisma or `lib/` functions
   - Serialize all `Date` → ISO strings
   - Render one `*Client.tsx` component with data as props
   - Example:
     ```typescript
     export const dynamic = 'force-dynamic'
     export default async function Page() {
       const data = await getData()
       return <ClientShell initialData={data} />
     }
     ```

2. **Client shell:** `components/<FeatureNameClient>.tsx` (PascalCase, starts with `'use client'`)
   - All state (hooks), event handlers, re-fetch logic
   - Initialize state directly from `initialData` prop (no `useEffect` data loading)
   - Example:
     ```typescript
     'use client'
     export default function HomeClient({ initialStats }) {
       const [stats, setStats] = useState(initialStats)
       // ... all interactivity here
     }
     ```

3. **Loading skeleton:** `app/<route>/loading.tsx` (optional; static server component)
   - Use `bg-surface-2 animate-pulse` exclusively
   - Mimic page layout for smooth transition
   - Shown only on client-side navigation, not first load

4. **Update Nav:** Add entry to `components/Nav.tsx` if this is a main route (top-level tab)

### New Database Model

1. **Schema:** Add to `prisma/schema.prisma`
   - Use `@id @default(cuid())` for primary keys
   - Use `DateTime @default(now()) @updatedAt` for timestamps
   - Add comments explaining purpose and relationships
   - Example:
     ```prisma
     model MyModel {
       id        String   @id @default(cuid())
       createdAt DateTime @default(now())
       updatedAt DateTime @updatedAt
       // ...relations, fields
     }
     ```

2. **Generate:** `npx prisma generate` (updates `app/generated/prisma/`)

3. **Turso DDL (if production):** 
   - Extract: `npx prisma migrate diff --from-empty --to-schema prisma/schema.prisma --script`
   - Apply: Via `@libsql/client` using a local script that reads `DATABASE_URL` from `.env`

4. **Data access:** 
   - For app settings: add getters/setters to `lib/settings.ts`
   - For data models: call Prisma directly in `lib/` functions or API routes

### New App Setting

1. **Concept:** App settings are persisted in the `Setting` model (key/value pairs)

2. **Implementation:**
   - Add getter/setter to `lib/settings.ts`
     ```typescript
     export async function getDailyGoal(): Promise<number> {
       const s = await prisma.setting.findUnique({ where: { key: 'dailyGoalSeconds' } })
       return s ? parseInt(s.value, 10) : DEFAULT_GOAL_SECONDS
     }
     ```
   - Call getter in `app/layout.tsx` if needed server-side (inject as CSS var or prop)
   - Wire GET/PUT in `app/api/settings/route.ts`
   - Add UI to `app/settings/page.tsx` form

3. **Rules:**
   - Always add getter + setter (not just direct Prisma calls)
   - Single source of truth in `lib/settings.ts`
   - Default values centralized, not scattered

### New Lib Module

1. **File:** `lib/<feature>.ts`
   - Pure, testable, no side effects
   - Export named functions (no default export)
   - Add JSDoc comments for public functions
   - Example:
     ```typescript
     /**
      * Compute X from Y.
      * @param input — the input
      * @returns the result
      * @note safe for server + client + tests; no impure calls
      */
     export function myFunction(input: string): string {
       // ...pure logic
     }
     ```

2. **Tests:** `tests/<feature>.test.ts` (Vitest)
   - Test business logic in isolation
   - No Prisma, no external APIs
   - Mock external dependencies
